# 本・図鑑・雑誌のデジタル化ワークフロー V4

> [!DATE] 2026-01-10
> [!PRIORITY] 品質 > 速度 > 量

このワークフローは、あらゆる種類の印刷物（図鑑、雑誌、コラム、学術書）を `ikimon.life` の Bio-Graph に取り込むための標準手順を定義します。

---

## 🚨 絶対遵守事項

### 1. API は使わない
*   `ikimon_digitizer_v3.php` や `VisionProcessor.php` などの **外部APIスクリプトは使用禁止**。
*   エージェント自身が `view_file` ツールで画像を直接閲覧し、人間のように内容を読み取ってJSONを作成する。
*   **理由**: あなたの「目」が最強のツールである。外部APIへの依存はエラーの温床となる。

### 2. 丁寧さが最優先
*   **1ページに時間をかけることは歓迎される。**
*   「早く終わらせよう」という発想を捨てる。
*   **ユーザーへの確認**: 不明点があれば必ず確認する。推測でデータを作らない。

### 3. 複数PC同期への配慮
*   このプロジェクトは **Google Drive** 経由で複数PCから編集される可能性がある。
*   **作業開始前**: 最新の状態であることを前提としない。ユーザーが明示的に「開始」を指示するまで待つ。
*   **書き込み時**: 大きなJSONファイルの編集は慎重に。競合が発生した場合はユーザーに報告する。
*   **一時ファイル**: 大量書き込みが必要な場合は、一時ファイルに書き込み、ユーザーの確認後にマージする。

## 📖 資料タイプ別の対応

### タイプ A: 図鑑（高密度プレート）
*   **例**: 学研の図鑑LIVE 昆虫、日本の野鳥650
*   **特徴**: 1ページに10〜30種が密集。
*   **必須手順**:
    1.  **Visual Census (視覚調査)**: JSON作成「前」に、ページ内の全図版を指差し確認で数える。
    2.  **Anti-Sampling (全数抽出)**: 数えた数だけJSONエントリを作成する。代表種だけを抜粋しない。
    3.  **Comparative Morphology (比較形態学)**: 「小さい」ではなく「〇〇より小さい」と記述。

### タイプ B: 雑誌記事・コラム
*   **例**: インセクタリゥム
*   **特徴**: 複数ページにまたがるフロー（P10の「用意」がP11の「育成」につながる等）。
*   **必須手順**:
    1.  **Flow Identification (フロー特定)**: 記事のページ範囲を最初に確認する（例: P10-P15がひとつの「カブトムシ飼育法」）。
    2.  **Cross-Page Context (文脈継承)**: JSONに `related_pages: [10, 11, 12, 13, 14, 15]` のようなメタデータを付与。
    3.  **Methodology Schema**: `rearing_methods`, `step_by_step_instructions` フィールドを使用。

### タイプ C: 索引・参考文献
*   **例**: 日本鳥類目録、学名索引
*   **特徴**: 生物エンティティはないが、権威付けや学名確認に重要。
*   **必須手順**:
    1.  **Index Validation (索引検証)**: 既存の Bio-Graph データの学名を索引と照合する目的で使用。
    2.  **エンティティ数はゼロ可**: `biological_entities: []` でも、`notes` や `taxonomic_group` で内容を記録する。

### タイプ D: 歴史書・文化書
*   **例**: 魚の文化史
*   **特徴**: 生物そのものよりも、民俗学・歴史的文脈がメイン。
*   **必須手順**:
    1.  **Entity Mining**: 「生物名」が出てきたらエンティティとして抽出。
    2.  **Cultural Context Tag**: `cultural_contexts` フィールドに時代背景やエピソードを格納。

## 📋 ステップバイステップ

### 0. 事前確認
1.  ユーザーから「〇〇の本を始めてください」という明確な指示を受ける。
2.  `book/{book_slug}/` に画像ファイルがあることを確認する。
3.  資料タイプ (A/B/C/D) を特定し、ユーザーに確認する。

### 1. Global Scan (全体俯瞰)
1.  ファイルリストを取得し、全体のページ数を把握する。
2.  サンプルページ（先頭、中盤、末尾）を閲覧し、密度パターンを把握する。
3.  `global_scan_report.md` または同等の計画メモを作成（任意）。

### 2. 単一ページ処理
1.  **Visual Census**: `view_file` でページを開き、**全ての**図版・ラベルを数える。
2.  **Count Declaration**: 「このページには X 種、Y 図版があります」とメモする。
3.  **JSON作成**: 数えた分だけエントリを作成。
4.  **Double Check**: 作成したJSONのエントリ数と、Step 2の宣言数が一致していることを確認。

### 3. バッチ完了後
1.  ユーザーに進捗を報告する（例: 「Pages 1-10 完了。合計 XX 種」）。
2.  次のバッチに進むか、ユーザーの指示を待つ。

### 4. 検証 (任意)
1.  作成したJSONファイルを開き、データの整合性を確認する。
2.  `status: "verified"` に更新する。

## 🚫 やってはいけないこと

1.  **APIキーを探す**: `ikimon_digitizer_v3.php` を実行しようとしない。
2.  **データを捏造する**: 見えない情報を「おそらくこうだろう」で埋めない。
3.  **ユーザー確認なしで大量処理**: 100ページを一気にやり切ろうとしない。区切りで報告する。
4.  **重複ファイルの乱立**: 同じデータを複数のJSONファイルに書き込まない。マスターJSONを1つに保つ。
