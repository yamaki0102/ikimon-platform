# platform_v2 日本語文章ガイド

更新日: 2026-04-22

## 0. この文書の位置づけ

- `platform_v2` の日本語コピー運用の正典
- `staging 用メモ` ではなく、public / trust / business / specialist / admin / ops の人間向け文言すべてに適用する基準
- 判定は「きれいに見えるか」ではなく、`何のページか / 何を返すか / 何を返さないか / 次に何をするか` がすぐ分かるかで行う

## 1. 読者理解

- 主対象は、一般ユーザー、初学者、難しい言葉が苦手な人
- 前提知識は置かない。`分類群` `同定` `公開範囲` のような言葉を使うなら、その場で意味が取れるようにする
- 団体相談面でも主語は担当者ひとりから始める。`自治体` `企業` を大きく語る前に、`どこで始めるか` `誰が使うか` `何を残すか` を先に置く

## 2. ページの役割

- 入口面: 何のページか、自分にどう関係あるか、次に何をするか
- 行動面: 何を入れるか、どこまでで十分か、あとでどう育てられるか
- 解説面: 理由、境界線、判断基準
- 信頼面: 何を扱うか、どこまで守るか、どこに連絡すればよいか
- 団体相談面: 誰向けか、何を無料で始められるか、何が相談対象か
- specialist / admin / ops: その画面で何を処理するのか、次の操作は何か、何が権限外か

## 3. 構文ルール

- 冒頭でページの役割を言う
- 見出しは意味が取れる文にする
- 1文に1つの仕事だけを書く
- 短くできるところは短くするが、説明が必要なページでは削りすぎない
- 同じ概念は同じ語で呼ぶ
- 例を早い位置に置く
- 限界や線引きを先に書く
- UI ラベルは操作語でそろえる

### 採用する書き方

- `散歩中に見つけた花や鳥を、場所つきで残せます。`
- `このページでは、どこまでを AI が助けて、どこからを人が決めるのかを説明します。`
- `まずは場所、日時、写真、ひとことメモがあれば十分です。`

### 採用しない書き方

- `日常が冒険に変わります。`
- `観察の価値を最大化します。`
- `AI があなたの観察体験を革新します。`

## 4. 人間味の信号

- 人が現場で使う名詞がある
- 何を助け、何を助けないかがある
- 具体例が 1 つでも入っている
- やりすぎない約束になっている
- 文の長さに少し強弱がある
- きれいすぎる対句より、意味の取りやすさを優先する

`人間らしさ` は、崩した文や誤字では作らない。  
`具体性` `責任境界` `例` `制約の開示` で作る。

## 5. AI臭さの禁則

次の症状が出たら、AI 生成っぽく感じられやすい。

- 抽象名詞だけの lead
- 対句で整えすぎた見出し
- 説明のないポジティブ断定
- 均一すぎる文長と段落
- AI を万能に見せる言い回し
- 誰が書いても同じに読める generic な比喩
- 内部語の直出し
- 英語ラベル混在
- SEO や更新頻度のための薄い総花ページ

### 禁止語・禁止パターン

- `place-first`
- `methodology`
- `authority-backed`
- `public claim`
- `readiness`
- `世界の解像度`
- `物語をガイド`
- `地図と AI を育てる`
- `AI が自動で決める`
- `常時分析して語りかける`

### production から移植しないもの

- 旧料金体系
- `Public 相当` `Personal` `Community` など旧プラン前提の説明
- AI 過剰訴求
- `ネイチャーポジティブ` を入口で広げすぎる導入
- 研究者、自治体、個人を 1 画面目で同列に混ぜる説明

## 6. 深度別テンプレート

### 入口面

- 1画面目で答える問い: `何のページか / 自分にどう関係あるか / 次に何をするか`
- 本文で答えること: 使う場面だけ
- 文量の目安: lead 2 文前後、カード本文 1〜2 文
- 失敗例: 世界観から入る、感情だけを煽る、操作名が分からない

### 行動面

- 1画面目で答える問い: `何を入れるか / どこまでで十分か`
- 本文で答えること: 最低限、あとで足せること、迷ったときの基準
- 文量の目安: 補助説明 2〜4 文
- 失敗例: 完璧な入力を要求する、専門用語でフォームを埋める

### 解説面

- 1画面目で答える問い: `このページで何が分かるか`
- 本文で答えること: 理由、線引き、次にどう使うか
- 文量の目安: H2 ごとに 2〜4 文
- 失敗例: リンク一覧だけ、結論だけ、良い話だけ

### 信頼面

- 1画面目で答える問い: `何を扱うか / どこまで守るか / どこに連絡するか`
- 本文で答えること: 取扱い、例外、問い合わせ先
- 文量の目安: 1項目 2〜5 文
- 失敗例: ぼかす、詩的にする、責任範囲を曖昧にする

### 団体相談面

- 1画面目で答える問い: `誰向けか / 何を無料で始められるか / 何を相談できるか`
- 本文で答えること: 対象、開始条件、相談前の準備、無料範囲と追加支援の境界
- 文量の目安: `対象 / 始め方 / 相談前` の 3 ブロック以上
- 失敗例: 売り文句から入る、巨大導入を前提にする

### specialist / admin / ops

- 1画面目で答える問い: `この画面で何を処理するか / どの権限で入っているか`
- 本文で答えること: 操作対象、必須項目、処理結果
- 文量の目安: 見出しは短く、フォームラベルは具体語
- 失敗例: 英語ラベル混在、内部語だけで進む、成功/失敗メッセージが曖昧

## 7. first screen promise と body obligation

- `first screen promise`
  - 1画面目で返す約束
  - 見出しと lead で表現する
- `body obligation`
  - 本文で果たす説明責任
  - 理由、線引き、例外、次の一歩を回収する

### done 行の基準

- lead は約束
- 本文は説明責任
- review matrix の `done` 行は `tokenized promise / obligation` で固定する
- `done` に上げる条件
  - lead と longform が更新済み
  - 対応 review doc がある
  - route 回帰と content guard が通る
  - staging 目視で promise と body がつながっている

## 8. specialist / admin / ops の例外

- 業務上必要な英語型名や API 名は UI に出さない
- ログや payload 名はそのままでも、ラベルは日本語にする
- `Grant` `Reject` `Audit` のような操作語は、日本語の操作として言い換える
- 迷わせないことを優先し、詩的な温度は入れない
