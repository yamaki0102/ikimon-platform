## 評価基準の定義

本レビューでは以下の評価基準を使用する：

| 評価 | 定義 |
|------|------|
| **S** | 外部リソース（法務レビュー完了、実reviewer確保、愛管ヒアリング完了、PoC実測値あり）を経た最高到達点。ドキュメントだけでは到達不可能 |
| **A** | Claude＋Codex（AIエージェント）だけでできる範囲の最高到達点。論理整合性、設計網羅性、実行計画の具体性、法務論点の特定が完璧 |
| **B** | 骨格は強いが、一部の設計が思想止まり、法務精度に甘さがある、運用モデルが抽象的 |
| **C** | ビジョンはあるが前提数字のズレ、設計の欠落、整合性の問題がある |

**A評価の条件（全て満たすこと）：**
1. 法務論点がAPPIネイティブに整理され、「ここは弁護士に確認要」が明確に切り出されている
2. Reviewer運営がrunbookレベル（Day 1チェックリスト、capacity計算、L3不在時フォールバック）まで具体化
3. Phase 6実証がfalsifiable（数値ゲート＋Go/No-Go判定＋2段階分離）
4. ミッション（地域・子ども・ウェルビーイング）と技術基盤（Evidence Tier、Canonical Schema）が「1つのコア基盤、2つの提供面」として統合
5. 供給能力がunit定義（site-month、reviewer-hour）され、無償提供の上限が逆算されている
6. 文書内の整合性ノイズがゼロ

**S評価に追加で必要なもの（Appendix D参照）：**
- 法務レビュー完了（弁護士によるAPPI 6論点の確認）
- 初期reviewer 10名の実在確保
- 愛管株式会社ヒアリング完了（「使えそうか？」の検証）
- BirdNET音響PoC G1-G7の実測値

## レビュー観点

### 1. A評価6条件の充足度チェック

各条件について「充足/部分充足/未充足」を判定し、根拠を示すこと。

### 2. v3.3/v3.4→v3.5の改善度

特に以下の新規要素を評価：
- Appendix D（法務レビュー依頼書）：6論点は適切に整理されているか？弁護士が効率的にレビューできる形式か？
- Reviewer Runbook（B7）：Day 1オンボーディング、L3不在時フォールバック、Capacity計算テンプレートは実行可能か？
- ミッションScorecard：3つの循環の測定指標は適切か？測定可能か？
- 744自治体の段階的cohort展開：初期2-3→拡大10-20→全国の移行条件は現実的か？
- テックニュース反映：SpeciesNet、Gemini Embedding 2（音声・動画直接埋め込み）、LiteRT、BirdNET Perch、DwC-DP、GX-ETS、ISSB BEESの戦略への統合は適切か？

### 3. 「1つのコア基盤、2つの提供面」の統合度

- Community面（子ども・地域・ウェルビーイング）とEnterprise面（OECM・TNFD・クレジット）は、技術的にも経営的にも1つのプロダクトとして成立しているか？
- ミッションScorecardの指標はCommunity面とEnterprise面の両方をカバーしているか？

### 4. テックニュース反映の戦略的妥当性

- Gemini Embedding 2のマルチモーダル埋め込み（音声80秒直接埋め込み）は、BirdNET音響PoC＋Canonical Schemaの設計にどう影響するか？正しく反映されているか？
- Google SpeciesNet（〜2,500種）とiNaturalist TFLite（〜500taxa）の使い分けは明確か？
- DwC-DP移行への対応は十分か？
- GX-ETS 4月義務化＋ISSB BEES 10月COP17のタイムラインは、Phase 6-7のロードマップと整合しているか？

### 5. 残存する懸念の網羅性

前回までの全レビューで指摘された懸念が全て対処（解決/切り出し/計画化）されているか？
