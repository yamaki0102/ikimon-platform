## B1. Evidence Tier 完全定義（Tier 1.5を含む）

| Tier | 名称 | 定義 | 生成方法 | 用途 |
|------|------|------|---------|------|
| **1** | Hypothesis-grade | AI候補のみ。人間未レビュー | パッシブセンシングのAI推論出力 | 内部解析、クエスト発行トリガー |
| **1.5** | Semi-validated | AI確信度が閾値以上（例：BirdNET ≥ 0.9）かつ、その種がその場所・時期に出現する生態学的妥当性がある | AI自動昇格。人間レビュー不要 | 統計処理、マップ表示（「要確認」マーク付き） |
| **2** | Review-grade | 1人以上のreviewer がoccurrenceを確認 | Trusted Reviewer の承認、または一般reviewer 1名の確認 | 図鑑表示、市民科学データ |
| **3** | Disclosure-grade | 複数reviewerの合意＋監査ログ＋samplingEffort記録 | 合意形成プロセス完了 | TNFD開示、OECM申請、GBIFへの公開 |
| **4** | Credit-grade | 第三者検証済み＋方法論準拠 | 外部監査プログラム | 生物多様性クレジット算出根拠 |

### Tier 1.5 の自動昇格条件

```
IF  AI_confidence ≥ threshold(species_group, site)
AND species IN known_species_list(location, season, habitat)
AND NOT rare_species_flag
THEN  auto_promote to Tier 1.5
       identificationMethod = "AI-auto-semi"
       identificationVerificationStatus = "semi-validated"
```

**閾値は分類群・サイト別に設定する。** BirdNET ≥ 0.9 は初期値であり、サイトごとの検証結果に基づいて調整する。

| 判定条件 | 実装方法 |
|---------|---------|
| AI_confidence閾値 | BirdNET初期値0.9。PoC実測後にサイト別・分類群別に調整 |
| 生態学的妥当性（known_species_list） | 地域チェックリスト（県レベルの既知種リスト）＋季節窓（月別の出現記録）で判定。初期は環境省レッドリスト＋都道府県別鳥類リストをベースに作成 |
| 希少種フラグ | 環境省レッドリスト＋都道府県版レッドリストに掲載されている種は自動フラグ。Tier 1.5に昇格させない |

**Tier 1.5は人間レビューの代替ではない。** 統計処理やマップ表示には使えるが、GBIFへの公開やTNFD開示にはTier 3以上が必要。reviewer工数を削減するためのトリアージ層。

## B2. Reviewer制度

### B2.1 Reviewer Tier

| レベル | 名称 | 条件 | 権限 |
|--------|------|------|------|
| **L0** | 一般ユーザー | アカウント登録のみ | Tier 1データへの同定提案。L1以上のreviewerが承認するまでTier 2に昇格しない |
| **L1** | Reviewer | 過去30件以上の同定実績＋正答率70%以上 | Tier 1→2の昇格承認。単独で1票 |
| **L2** | Trusted Reviewer | 過去100件以上＋正答率85%以上＋特定分類群の専門性 | Tier 1→2を単独承認。Tier 2→3の合意形成に参加 |
| **L3** | Expert Reviewer | 分類学の専門家（論文・図鑑著者等）。運営が招待 | Tier 2→3の決裁権。dispute resolutionの最終判断 |

### B2.2 レピュテーションスコア

```
reputation_score = (agreed_identifications / total_identifications) × weight_factor

weight_factor:
  - 難易度の高い同定（AI確信度 < 0.5）: ×2.0
  - 希少種の同定: ×1.5
  - dispute解決への貢献: ×1.5
  - 誤同定（後にoverrideされた）: ×0.5（ペナルティ）
```

### B2.3 昇格・降格条件

| 遷移 | 条件 |
|------|------|
| L0 → L1 | 30件以上の同定提案＋うち70%以上がL1以上に承認された |
| L1 → L2 | 100件以上＋正答率85%＋特定分類群での一貫した実績 |
| L2 → L3 | 運営招待のみ。分類学の専門家 |
| L1 → L0（降格） | 直近50件の正答率が50%を下回った場合 |
| L2 → L1（降格） | 直近100件の正答率が70%を下回った場合 |

## B3. 合意形成プロセス（Tier 2→3）

### B3.1 クォーラム（定足数）

| 条件 | 必要票数 |
|------|---------|
| 通常の種（一般的） | L1以上 2票の合意 |
| 希少種・保全上重要な種 | L2以上 2票の合意、またはL3 1票 |
| 同定が困難な分類群（蘚苔類、微小甲虫等） | L2以上 2票の合意＋根拠文献の引用 |

### B3.2 合意形成フロー

```
Tier 2 occurrence
    ↓
reviewer A が同定を提出（scientificName, 根拠）
    ↓
reviewer B が確認
    ├── 合意 → Tier 3に昇格。監査ログ記録
    └── 不合意 → Dispute に移行
```

### B3.3 Dispute Resolution

```
Dispute 発生（reviewer間で同定が分かれた）
    ↓
Step 1: 根拠の提示（写真の指摘箇所、文献引用、比較画像）
    ↓ 48時間以内
Step 2: 3人目のreviewer（L2以上）を招集
    ↓
Step 3: 多数決
    ├── 合意形成 → Tier 3に昇格
    └── 合意不成立 → L3 Expert Reviewer にエスカレーション
         ├── Expert判断 → Tier 3に昇格（Expert判断として記録）
         └── 判断不能 → Tier 2に留まる。「要追加証拠」としてクエスト発行
```

**disputeの平均解決目標：** 72時間以内

### B4.1 獲得チャネル

|---------|------|---------|-----------|

## B5. 月次KPI

| KPI | 目標値（PoC期間） | 測定方法 |
|-----|-----------------|---------|
| Active Reviewer数 | 10名以上 | 月に1件以上のレビューを行ったreviewer数 |
| Tier 1→2 昇格率 | 30%以上 | 月間のTier 1イベント数に対するTier 2到達数 |
| Tier 1→2 所要時間 | 72時間以内（中央値） | Tier 1生成からTier 2承認までの時間 |
| Tier 2→3 合意率 | 80%以上 | Tier 2からTier 3に至った割合 |
| Dispute発生率 | 10%以下 | Tier 2→3プロセスでdisputeになった割合 |
| Dispute解決時間 | 72時間以内（中央値） | dispute発生から解決までの時間 |
| reviewer離脱率 | 月20%以下 | 前月activeだったreviewerが当月inactiveになった率 |
| 1 review あたりの工数 | 3分以下（中央値） | reviewer操作ログから算出 |

## B6. Reviewer UX設計原則

- **レビューキューは5分で終わる単位に区切る。** 通勤電車でできるくらい手軽に
- BirdNET結果は波形＋スペクトログラム＋AI候補を表示。reviewerは「合ってる/違う/わからない」の3択
- 画像は同定に必要な部位のハイライト表示（AIが同定根拠として注目した箇所）
- 「わからない」は恥ではない。スキップのペナルティなし
- 分類群ごとのフィルタ。自分が得意な分類群だけレビューできる

## B7. Reviewer Runbook（Day 1 オンボーディング）

### B7.1 新規Reviewer（L0）の初日チェックリスト

```
□ 1. ikimon.lifeアカウント作成（メール認証）
□ 2. プロフィール設定（得意な分類群を選択：鳥類/植物/昆虫/哺乳類/その他）
□ 3. チュートリアル完了（5分。BirdNET結果の見方、承認/棄却/スキップの操作）
□ 4. 練習レビュー3件（正解付き。フィードバックあり）
□ 5. 初回レビュー5件（実データ。L1以上がバックグラウンドで検証）
```

### B7.2 L0→L1 昇格までのガイド

| ステップ | 目安期間 | 内容 |
|---------|---------|------|
| 練習期 | 1-2週間 | 毎日5-10件のレビュー。得意な分類群に集中 |
| 実績積み上げ | 2-4週間 | 30件到達を目指す。正答率70%以上を維持 |
| L1昇格 | 累計30件＋正答率70% | 自動昇格。通知あり |

### B7.3 Dispute発生時のエスカレーション手順

```
1. Disputeが発生（reviewerAとBで同定が分かれた）
   → システムが自動で「Dispute」フラグを付与
   → 両者に通知：「根拠を追記してください（48時間以内）」

2. 48時間経過
   → L2以上のreviewerに自動エスカレーション
   → 3人目のreviewerがレビューキューに入る

3. 3人目がレビュー
   → 多数決で解決 → Tier 3に昇格
   → 3者とも異なる場合 → L3 Expert Reviewerにエスカレーション

4. L3不在時のフォールバック（v3.5で追加）
   → 72時間以内にL3が対応できない場合
   → 「要追加証拠」としてTier 2に留め置き
   → クエストを発行して追加証拠を募集
   → L3確保後に再レビュー
```

### B7.4 Reviewer Capacity計算テンプレート

```
月間処理能力 = Active Reviewer数 × 平均review件数/人/月

例（PoC初期）：
  Active Reviewer: 10人
  平均 review: 300件/人/月（3分/件 × 30分/日 × 20日 の控えめ想定）
  月間処理能力: 3,000件

  Tier 1生成量: 500件/site-month
  Tier 1.5自動昇格: 50%（250件は人間レビュー不要）
  要レビュー件数: 250件/site-month

  対応可能サイト数: 3,000 ÷ 250 = 12 site-months
```

**KPIは人数ではなく処理能力で追う。** 月次で `Active Reviewer数 × 平均件数/人` を計算し、`要レビュー件数の合計` と比較。capacity utilization > 80% になったらreviewer増員アラート。
