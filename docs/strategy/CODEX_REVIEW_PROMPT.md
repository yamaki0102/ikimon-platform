# Codex レビュー＆拡張プロンプト

> このプロンプトを Codex に渡して、独立した視点からの改善提案を得る。

---

## プロンプト

```
あなたは ikimon.life の技術アドバイザーです。

ikimon.life は市民参加型の生物多様性プラットフォームで、3つの観察モード（写真投稿、ウォーク音声検出、ライブスキャン）を持ちます。今日、Phase 6 PoC の初日フィールドテストを実施しました。

添付の `IMPROVEMENT_IDEAS_OPUS.md` に Claude Opus 4.6 が出した10の改善提案と、フィールドテストの結果、現在のアーキテクチャ図があります。

以下のタスクを実行してください:

## 1. Opus の提案への批判的レビュー
- 各提案の実現可能性、優先度、リスクを評価
- 「これは今やるべきではない」があれば理由と共に指摘
- 「これは提案以上に重要」があれば強調

## 2. Opus が見落としている改善提案（5つ以上）
- Opus とは異なる視点からの提案を期待
- 特に: UX、データ品質、スケーラビリティ、コミュニティ形成、法務/倫理
- 「市民科学プラットフォームとして」の視点を重視

## 3. 100年後の視点
- 100年後にこのデータが科学的に使われるとして、今何を記録すべきか
- 現在のデータスキーマ（Canonical Schema）に欠けているフィールドはあるか
- データの長期保存戦略（フォーマット、冗長性、アクセシビリティ）

## 4. 気をつけるべき落とし穴
- 「市民科学データの品質問題」の先行事例から学べること
- iNaturalist, eBird, Merlin 等の既存プラットフォームの教訓
- 日本特有の法的・文化的考慮事項

## 5. 実装優先度マトリクス
- 提案を「影響度 × 実装コスト」でマトリクスに配置
- 「今週やるべき」「今月やるべき」「次四半期」「将来検討」に分類

## 制約
- 技術スタック: PHP 8.2 + SQLite + Alpine.js(→vanilla JS移行中) + MapLibre + Gemini API + BirdNET on VPS
- ホスティング: Xserver VPS 12GB RAM Ubuntu 24.04
- 予算: Gemini API 月 ~$30 以内
- ユーザー規模: PoC 段階、5-10人 → 100人を目標

## 出力形式
- Markdown で構造化
- 各提案に ID を振る（参照しやすいように）
- 具体的な実装方針があれば、ファイル名とコードの方向性も記述
```

---

## 使い方

1. Codex のセッションを開く
2. このプロンプトと `IMPROVEMENT_IDEAS_OPUS.md` を渡す
3. `DEVELOPMENT_PLAN.md` と `CLAUDE.md` も参照させる
4. 出力を `docs/strategy/IMPROVEMENT_IDEAS_CODEX.md` に保存
5. 両方のアイデアを統合して次のスプリント計画に反映
