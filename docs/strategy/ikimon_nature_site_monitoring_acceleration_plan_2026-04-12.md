## 0. この計画の一言要約

ikimon の企業向け価値を、`生物多様性の見える化` から

**`自然共生サイトのモニタリングを、すぐ・安く・迷わず始められる基盤`**

へ再定義する。

この計画の目的は、全体リニューアルの中で企業向け導線を

- 重いコンサル型
- 手作業の立ち上げ型
- GIS 前提の専門家型

から切り離し、

- `エリア候補がすぐ出る`
- `初回観察会がすぐ立つ`
- `無料で試せる`
- `担当者1人でも回し始められる`

状態まで持っていくことにある。

---

## 1. 解くべき本当の問題

企業が困っているのは、観察そのものより前段である。

1. 対象エリアをどう決めるかわからない
2. 境界や観察単位をどう作るかわからない
3. 初回観察会をどう設計するかわからない
4. 記録を何に使えるかが見えない
5. 外注しないと始められないと思っている

つまり競争相手は `他の見える化マップ` ではない。

競争相手は次の 3 つである。

- 手作業の立ち上げコスト
- 社内で止まる検討コスト
- 調査会社に投げる前提の高コスト構造

---

## 2. 勝ち筋

主案は次で固定する。

**`自然共生サイト 即時起動パッケージ`**

企業は住所、拠点名、敷地名、既存サイト名のいずれかを入力するだけで、
ikimon が以下を返す。

1. 監視対象エリア候補
2. 初回観察会 2 件
3. 記録開始導線
4. 初回共有用サマリー

これにより、企業向けの価値訴求を

- `可視化できます`
- `データが残せます`

ではなく、

- `モニタリングがすぐ始まります`
- `担当者の準備工数が大幅に減ります`
- `無料で試せます`

へ移す。

---

## 3. プロダクト約束

企業向けの約束は次の 3 本に絞る。

### 3.1 Primary promise

**`自然共生サイトのモニタリングを、すぐ始められる。`**

### 3.2 Secondary promise

**`エリア設定から観察会準備までを、省力化できる。`**

### 3.3 Entry offer

**`まずは無料で試せる。`**

補足:

- `すごい` `革新的` のような抽象形容は避ける
- `見える化` は補助便益であり主訴求にしない
- `専門家がいなくても初速が出る` を重視する

---

## 4. 全体リニューアルとの接続

この計画は、全体リニューアル方針と矛盾させない。

### 4.1 表の顔

- 個人向け: place-first product
- public hero: 日常の自然体験

### 4.2 裏の収益導線

- 企業向け: 自然共生サイトの高速起動
- backstage: `for-business/` と corporate workspace

### 4.3 整合の取り方

- public top は企業色を強くしすぎない
- 企業導線は footer / secondary CTA / dedicated LP に置く
- 企業向けで作った `place / event / monitoring` 機能は、個人向け place-first 体験にも還元する

---

## 5. 著作権・行政リレーション・ガードレール

この計画は、`速く作ること` より先に `揉めない構造` を満たす。

### 5.1 著作権とデータ利用の原則

- 環境省、自治体、研究機関、地図事業者、外部DBのデータは `出典・利用条件・再配布可否` を確認せずに商用再利用しない
- 外部データは `表示参照` と `ikimon 独自生成データ` を分離する
- 他者データを ikimon の生成物のように見せない
- GeoJSON、境界線、航空写真、地図タイル、レッドリスト情報、保護地域情報は、ソースごとに利用条件台帳を持つ
- コピー、図版、説明文は環境省等の文章を要約前提で扱い、転載前提で作らない

### 5.2 環境省から嫌われないための運用原則

- `環境省公認` `環境省準拠` のような誤認を招く表現は使わない
- 行政施策は `参照` するが `代理執行` のように見せない
- 自然共生サイト制度の説明は、制度紹介と ikimon の実務支援を明確に分ける
- 希少種やセンシティブ情報は公開精度を制御し、保全を損なう UX を作らない
- 行政・研究・保全団体に対しては `対立的な置き換え` でなく `実務を軽くする補助基盤` として位置づける

### 5.3 プロダクト表現の禁止事項

- `公式データベースになる` と断定しない
- `申請がこのまま通る` と保証しない
- `専門家不要` と言い切らない
- `AI が正しい境界や正しい調査設計を確定する` と言わない
- 行政名や制度名を借りて権威づけしない

### 5.4 実装前に必要な最低成果物

- 外部データソース一覧
- ソースごとの利用条件メモ
- 表示上の attribution 方針
- 企業向けコピーの法務・広報レビュー
- 希少種 / センシティブ地点の公開ルール

### 5.5 立ち位置

ikimon の立ち位置は次で固定する。

**`制度を代替するプロダクト` ではなく、制度対応や継続モニタリングの実務負荷を減らす補助基盤`**

## 7. 新しい企業向け体験

企業向け体験は次の 5 ステップに統一する。

### Step 1. 拠点入力

入力:

- 住所
- 拠点名
- 施設名
- 任意で地図上の位置指定

返すもの:

- site 候補
- 周辺環境の要約
- 自動推定した監視対象エリア候補

### Step 2. エリア確認

返すもの:

- 境界候補 1-3 件
- 想定面積
- 監視上の説明文

ルール:

- 完全自動が無理でも 80% の候補をすぐ返す
- 手修正は許容する
- 最初から GIS 専門操作を要求しない

### Step 3. 初回観察会自動作成

自動で作るもの:

- 観察会 A: 朝 or 日中の基本コース
- 観察会 B: 夕方 or 別分類群コース

最低含むもの:

- タイトル
- 対象エリア
- 目安時間
- 推奨分類群
- 目標記録数
- 持ち物
- 参加導線

### Step 4. 記録開始

必要条件:

- 参加リンク発行
- スマホからすぐ記録開始
- 初回は rough label で保存可能

### Step 5. 初回レポート

返すもの:

- 実施したこと
- 記録件数
- 種数
- 継続時の価値
- 次回提案

## 8. 商品設計

### 8.1 無料トライアル

無料枠は次のように定義する。

- 1 拠点まで
- エリア候補生成
- 初回観察会 2 件自動作成
- 記録開始
- 初回サマリー表示

無料でやらないもの:

- カスタム分析
- 継続レポート自動配信
- 複数拠点横断管理
- 詳細 TNFD / CSR 向け出力

### 8.2 有料で効かせる価値

- 複数拠点管理
- 継続モニタリング
- 担当者ワークスペース
- レポート出力
- 自治体 / 社外説明向け整形
- 観察会テンプレの高度化

## 9. 実装フェーズ

## Phase 0. 定義固定 + コンプライアンス前提確認

目的:

- 企業向け訴求を `可視化` から `高速起動` へ切り替える

決めるもの:

- primary copy
- 無料枠の範囲
- 企業向け CTA
- エリア候補の入力仕様
- 外部データソースの利用条件
- attribution の出し方
- 行政誤認を防ぐ表現ルール

完了条件:

- LP と営業で同じ一文を使える
- `無料で何ができるか` が曖昧でない
- 外部データの扱いが未確認のまま進まない

## Phase 1. 企業LPの再定義

対象:

- `for-business/index.php`
- 関連 CTA

目的:

- モニタリング省力化の訴求を固定する

必須要素:

1. `すぐ始められる`
2. `省力化できる`
3. `無料で試せる`
4. `エリア設定 -> 観察会 -> 記録 -> 初回共有` の流れ
5. `制度紹介` と `ikimon が肩代わりする実務` の分離

完了条件:

- 初見で `何が速いのか` が伝わる
- 見える化サービスに見えすぎない

## Phase 2. Site Quickstart Wizard

対象:

- site 作成導線
- `site_editor.php`
- 関連 API

目的:

- `住所入力 -> エリア候補生成` を最短導線で成立させる

実装:

- 拠点入力フォーム
- 候補エリア返却
- 手修正 UI
- site 保存

完了条件:

- 担当者が迷わず site を 1 件作れる
- GeoJSON の知識が不要

## Phase 3. Auto Event Bootstrap

対象:

- `create_event.php`
- event 自動作成ロジック
- `survey.php`

目的:

- site 作成直後に初回観察会 2 件を立てられるようにする

実装:

- 初回イベントテンプレ
- 時間帯別テンプレ
- 推奨分類群の自動付与
- 招待リンクの即時発行

完了条件:

- site 作成後 5 分以内に 2 件の観察会が存在する

## Phase 4. Monitoring Workspace

対象:

- `corporate_dashboard.php`
- `dashboard_portfolio.php`
- `dashboard_municipality.php`

目的:

- 監視の開始後に `次に何をするか` を返す

見るべきもの:

- 実施済み観察会
- 未調査エリア
- 次回候補
- 継続提案

完了条件:

- ダッシュボードが集計画面でなく運用画面になっている

## Phase 5. Proof and Expansion

対象:

- `csr_showcase.php`
- `generate_*_report.php`

目的:

- 企業が社内外に説明しやすい状態を作る

完了条件:

- `無料で試した後、なぜ継続費を払うのか` が説明できる

## 10. 優先順位

実装順は次に固定する。

1. `for-business/index.php` の訴求修正
2. site quickstart 導線
3. エリア候補生成
4. 観察会 2 件自動作成
5. 無料トライアル導線
6. corporate dashboard の `次アクション化`
7. report / showcase 接続

理由:

- コピーが曖昧なまま機能を作るとズレる
- まず最初の 10 分価値を作るべき
- ダッシュボード磨きは立ち上げ自動化の後でよい

## 11. LP / 営業メッセージ

第一候補:

**`自然共生サイトのモニタリングを、すぐ始められる。`**

補助:

- `エリア設定から観察会の立ち上げまで、ikimon が省力化します。`
- `まずは無料で試せます。`

第二候補:

**`自然共生サイトの立ち上げを、最短で。`**

補助:

- `対象エリアの設定、初回観察会、記録開始までを一気に。`
- `担当者1人でも始めやすい設計です。`

禁止:

- `見える化` を hero の主語にしない
- `AI が全部やる` と言いすぎない
- `専門家不要` と断定しすぎない
- `環境省公認` のような誤認表現を使わない

## 13. リスクと対策

## Risk 1. 単なるマップLPに見える

対策:

- hero を `起動速度` に固定する
- 地図は補助説明に下げる

## Risk 2. 自動境界の精度が低い

対策:

- `候補生成 + 手修正` を前提にする
- 完全自動を約束しない

## Risk 3. 観察会自動生成が薄い

対策:

- 最初は 2 テンプレに限定する
- 時間帯と分類群だけでも十分価値にする

## Risk 4. 無料範囲が広すぎて商売にならない

対策:

- `初回起動まで無料`
- `継続運用から有料`
  に明確に切る

## Risk 5. 著作権・利用条件・行政表現で事故る

対策:

- 外部データを使う画面ごとに出典と利用条件を棚卸しする
- 環境省等の制度説明と ikimon 独自機能を分離表記する
- コピー公開前に法務 / 広報 / 制度誤認チェックを通す

---

## 14. 今すぐ固定する意思決定

以下は先に固定しないと実装がぶれる。

1. 企業向け primary value は `見える化` ではなく `高速起動`
2. 無料枠は `1拠点 + 初回観察会2件 + 初回サマリー` まで
3. エリア確定は `完全自動` でなく `候補提示 + 補正`
4. 初回観察会は `自動で2件` を商品事実にする
5. corporate dashboard の主役は `集計` でなく `次に何をするか`
6. 行政名・制度名は権威づけでなく参照情報として扱う
7. 外部データ利用条件が未確認の機能は公開しない

---

## 15. この計画の本質

この計画は、企業向けページを1枚増やす話ではない。

本質は、

**`自然共生サイトのモニタリング開始コストを、プロダクトの力で構造的に下げること`**

にある。

ikimon が勝つべき場所は、

- 生物多様性をただ見せること
- きれいな地図を作ること

ではなく、

- `明日から始められること`
- `担当者の工数を減らせること`
- `外注前提を崩せること`

である。
