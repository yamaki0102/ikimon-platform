# Phase 18: 生態系インテリジェンス — 地理データ × AI 統合ロードマップ

## ビジョン

> スキャンするたびに、その場所の生態系が「わかる」ようになる。
> 使えば使うほど情報が濃くなり、精度が高まる。
> 市民が楽しくやった結果が、科学的に価値のあるデータになる。

## 実装ロードマップ

### Sprint A: データ基盤（1-2日）

**目的**: 国交省 + OSM データを取り込んで、クエリできる状態にする

1. **国交省 土地利用メッシュ取得**
   - 国土数値情報ダウンロードサービスから浜松市周辺を取得
   - GeoJSON 変換 → SQLite に格納
   - カテゴリ: 田, 畑, 森林, 荒地, 建物用地, 道路, 鉄道, その他用地, 河川湖沼, 海浜, 海水域, ゴルフ場

2. **国交省 河川データ取得**
   - 河川中心線 + 河川区域
   - 「最寄りの河川までの距離」を計算できるように

3. **OSM データ取得**
   - Overpass API で浜松市周辺の公園/水系/遊歩道/緑地を取得
   - GeoJSON → SQLite

4. **GeoContext API 新規作成**
   - `/api/v2/geo_context.php`
   - 入力: lat, lng
   - 出力: { land_use, nearest_water_m, nearest_park, elevation, green_ratio, trails[] }

### Sprint B: 環境文脈エンジン（1日）

**目的**: 検出に「なぜここにいるか」を付与

1. **scan_classify.php のプロンプトに環境文脈を注入**
   - GeoContext の結果をプロンプトに追加
   - 「この地点は河川から200m、落葉広葉樹林」→ Gemini がより正確に同定

2. **session_recap.php の貢献メッセージに環境データ**
   - 「この地点は都市公園内。都市の生物多様性記録として特に価値」
   - 「河川沿いの記録。水辺生態系のモニタリングに貢献」

3. **検出バナーに環境アイコン**
   - 🏞️ 公園内 / 🌊 水辺 / 🌲 森林 / 🏙️ 市街地

### Sprint C: 出現予測エンジン（2-3日）

**目的**: 「次に出会える生物」をスキャン前に表示

1. **予測モデル構築**
   - 入力: 場所(lat,lng), 月, 時間帯, 土地利用タイプ
   - 出力: [{ species, probability, last_seen, habitat_match }]
   - 手法: OmoikaneDB の季節・生息地情報 + 過去の検出実績 + Embedding類似検索

2. **予測API**
   - `/api/v2/predict_species.php`
   - Gemini Embedding 2 で「この環境に似た場所で見つかった種」を検索

3. **UI（field_scan.php/walk.php の開始前画面）**
   - 「この場所で出会えるかも」リスト（3-5種）
   - 種カード形式（和名 + へぇポイント + 出現確率）

### Sprint D: ルート推薦 + ポテンシャルマップ（2-3日）

**目的**: 「今日どこを歩けばいいか」の提案

1. **ポテンシャルスコア計算**
   - 土地利用多様性 × 水辺近接度 × 緑地率 × 既存データ密度の逆数
   - 「データが少ない高ポテンシャル地点」を優先

2. **ルート推薦**
   - OSM の遊歩道ネットワーク上で、ポテンシャルスコアの合計が最大のルートを推薦
   - explore.php に「おすすめルート」セクション

3. **ポテンシャルマップ**
   - MapLibre GL JS でヒートマップ表示
   - explore.php に統合

### Sprint E: 図鑑AI検索 + スキャン比較（1-2日）

**目的**: 「これ何？」に答える + 時系列の学び

1. **自然言語 → 種検索**
   - Gemini Embedding 2 でクエリをベクトル化
   - OmoikaneDB の形態・生息地・季節情報と類似検索
   - zukan.php に自然言語検索バー追加

2. **スキャン比較**
   - 同じ場所の過去スキャンと現在を自動比較
   - 新出現種/消失種をハイライト
   - session_recap.php に「前回との変化」セクション

## データフロー詳細

### 「なぜここにいるか」の生成フロー

```
検出「カワセミ」(field_scan.php)
    ↓
GeoContext API (lat, lng)
    ↓ 返却: { nearest_water: "天竜川支流 150m", land_use: "河川敷", green_ratio: 0.7 }
    ↓
OmoikaneDB 検索 (カワセミの生息環境)
    ↓ 返却: { habitat: "清流・河川", diet: "小魚", season: "留鳥" }
    ↓
Gemini Flash Lite で文生成:
    「カワセミは清流のハンター。近くの天竜川支流（150m）が狩場です。
     ホバリングから一気にダイブして小魚を捕る姿は圧巻。留鳥なので年中観察可能。」
    ↓
バナー + session_recap に表示
```

### 予測の学習ループ

```
Sprint C 初期: OmoikaneDB の季節情報のみ → 粗い予測
    ↓ ユーザーがスキャン
Sprint C+1ヶ月: 実データで予測を補正 → 精度向上
    ↓ さらにスキャン蓄積
Sprint C+3ヶ月: 場所×季節×天候の十分なデータ → 高精度予測
    ↓
「使えば使うほど精度が上がる」体験の実現
```

## 優先度と依存関係

```
Sprint A (データ基盤) ←── 全ての前提
    ↓
Sprint B (環境文脈) ←── A に依存、最も即効性が高い
    ↓
Sprint C (出現予測) ←── A + 既存データに依存
Sprint D (ルート推薦) ←── A に依存
Sprint E (AI検索) ←── Embedding 2 のみ依存（独立）
```

**推奨実行順**: A → B → E → C → D

Sprint E は A に依存しないので、A と並行可能。
