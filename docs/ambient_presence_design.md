## なぜ「つながり」ではなく「存在感」なのか

SNSは「つながり」を売る。フォロー、いいね、コメント、シェア。
でもそれは**会話を強制する**。返事をしないと「無視した」になる。

ikimonが目指すのは違う。

**朝の散歩で、見知らぬ人とすれ違う。**
目も合わせない。名前も知らない。
でも「この道を歩いているのは自分だけじゃない」と知っている。
それだけで、少しだけ安心する。

これが **Ambient Presence（環境的存在感）** 。

## インスピレーション — 3つのゲームが証明したこと

### 🎮 Death Stranding — 「誰かが置いたはしご」

荒野を一人で歩いている。崖に差しかかる。
降りられない。そう思った瞬間、崖にはしごがかかっている。
名前のない誰かが、いつか置いたものだ。

ボタンを押す。「いいね」。
置いた人がどこにいるか知らない。今もプレイしているか分からない。
でも「ありがとう」という感情は本物だ。

**ikimonへの翻訳**:
- 地図上の匿名の観察ピンが、誰かの「はしご」
- 「このエリアに誰かがいた」という事実だけで充分
- 自分の投稿も、いつか誰かの「はしご」になる

**なぜこれが効くのか（心理学）**:

> 社会的促進（Social Facilitation）: 他者の存在を認識するだけで、自身の行動が促進される。
> この効果は、相手が見知らぬ人でも、匿名でも、発生する。
> — Zajonc, 1965

つまり、**名前も顔も要らない。「いる」と知るだけで、人は動く。**

---

### 🎮 Detroit: Become Human（以下DBH）— 「あなたと同じ道を選んだ人: 34%」

章が終わる。画面に分岐チャートが広がる。
自分が辿ったルートが光っている。
そして各分岐点に、小さな数字。

「左を選んだ人: 67%」「右を選んだ人: 33%」

自分は多数派だった。あるいは少数派だった。
正解でも不正解でもない。ただ、**自分の選択が全体の中に位置づけられる。**

**「実績」との決定的な違い**:
- 実績: 「あなたは100回投稿しました 🏆」→ 自分の話。閉じている
- DBH: 「あなたと同じ判断をした人: 34%」→ 世界に開いている

実績は**自己完結**する。DBHのチャートは**他者の存在を呼び起こす**。

**ikimonへの翻訳**:
- ID Wizardのフローチャートに選択統計を載せる
- 同定結果に「同じ判断をした人の割合」を表示
- 種の観察タイムラインに「自分の発見が全体のどこにあるか」

**なぜこれが効くのか（心理学）**:

> 社会的証明（Social Proof）: 人は自分の判断が他者と一致しているか知ることで安心する。
> しかしこの効果は、直接的な社会的圧力がない状態でこそ最も純粋に機能する。
> — Cialdini, 1984

つまり、**「みんながやってるからやろう」ではなく、「やった後に、みんなもやってたと知る」。** 順番が逆。

### ⚠️ 有限ゲーム vs 無限ゲーム — DBHとikimonの決定的な違い

DBHは**エンディングがあるゲーム**。章が終わり、物語が閉じ、クレジットが流れる。
フローチャートが機能するのは、**「ここが終わり」が明確だから**。
プレイヤーは振り返る余裕がある。「あの選択は正しかったのか？」

ikimonにエンディングはない。種は無限に存在し、季節は巡り、地図は永遠に埋まらない。
チャートを「全体像」として見せると、**果てしなく広がり続ける未完成の図**になる。
それは「達成感」ではなく「虚無感」を生む。

**解決: 自然のリズムが「章」を作る — ただし四季は前提にしない**

DBH の「章の終わり」を、ikimon では**自然の区切り**に置き換える。
ただし**四季は温帯の特権**。熱帯に乾季/雨季しかない国、赤道直下で季節変動がほぼない国もある。
「春が来ました」を全世界に送ったら、シンガポールのユーザーは意味不明になる。

**原則: 季節はハードコードしない。緯度帯に応じた自然区分を使う。**

| 緯度帯 | 自然の区切り | 章の単位 |
|---|---|---|
| **温帯** (日本, 欧州, 北米) | 四季 (春夏秋冬) | 3ヶ月ごとの季節サマリー |
| **熱帯** (東南アジア, 中南米) | 雨季/乾季 | 雨季サマリー / 乾季サマリー |
| **赤道直下** (シンガポール, 赤道ギニア) | 季節差ほぼなし | **四半期** (Jan-Mar, Apr-Jun, ...) |
| **南半球** (オーストラリア, 南米南部) | 四季（北半球と逆） | 反転した3ヶ月サマリー |

**実装方針 — 3層チャプターシステム**:

```
  Layer 1 — ミクロ章: 1回の同定体験（常にどこでも成立）
  Layer 2 — メソ章: 地域の自然区分に基づくサマリー
  Layer 3 — マクロ章: 「あなたの2026年」年間まとめ
```

Layer 2 の決定ロジック:
```
  if (ユーザーの観察地点の緯度 が 23.4°N〜23.4°S) {
    // 熱帯: 季節名を使わない
    chapter = "Jan–Jun / Jul–Dec" または地域の雨季/乾季
  } else {
    // 温帯: 季節名あり。南半球なら反転
    chapter = ユーザーのロケールに応じた季節名
  }
```

**言語と季節名の扱い**:
- 「春」「Summer」「Musim Hujan（雨季）」のように、ロケール依存の季節名を使う
- 季節名が存在しない地域では、期間表示のみ（「2026年 1月〜6月の記録」）
- **絶対に「Spring」を全世界に強制しない**

温帯の例:
```
  ── 2026年 春の記録（3月〜5月）──
  あなたは 12種を記録しました
  静岡市の完成度を +4% 進めました
  あなたと同じ時期に観察した人: 89人
```

熱帯の例:
```
  ── 2026年 雨季の記録（5月〜10月）──
  あなたは 8種を記録しました
  Chiang Maiの完成度を +2% 進めました
  あなたと同じ時期に観察した人: 34人
```

赤道直下の例:
```
  ── 2026年 前半の記録（1月〜6月）──
  あなたは 15種を記録しました
  Singaporeの完成度を +1% 進めました
  あなたと同じ時期に観察した人: 127人
```

**これがikimon版の「章末チャート」。**
- 物語を**ユーザーが選んだ**のではなく、**自然（または時間）が閉じた**
- ゲームには人工的な区切りが必要。ikimonは**その土地の自然がそれを提供する**
- 四季がない場所でも、時間は等しく流れる。時間そのものが最も普遍的な「章」

同定ごとのフローチャート（装置3）は、DBHの「1章」に相当する。
- 1回の同定体験 = 始まり（写真を見る）→ 分岐（特徴を選ぶ）→ 終わり（種名が決まる）
- **有限の構造を持つ。** だからフローチャートが成立する。**緯度に関係なく、世界中どこでも同じ構造**
- 季節/期間サマリーは「全章を振り返るエンディング画面」に相当する

### 🎮 Journey — 「名前のない旅の仲間」

砂漠を一人で歩いている。
ふと、遠くに光が見える。近づくと、もう一人のプレイヤーがいる。
名前は分からない。話せない。できるのは「チャイム」を鳴らすだけ。

一緒に歩く。崖を登る。嵐を越える。
最後に、その人とは離れ離れになる。

エンドロールに、その人のIDが一瞬だけ表示される。
それだけ。でもプレイヤーは泣く。

**ikimonへの翻訳**:
- 地図で、同じエリアで同じ日に観察した「誰か」の存在を暗示する
- 「今日、このエリアであなた以外に2人が観察しました」
- 直接会えない。話せない。でも「同じ空の下にいた」

**なぜこれが効くのか（心理学）**:

> 脱人格化された共感（Depersonalized Empathy）:
> 最小限の情報（存在と行動のみ）は、かえって相手への想像力を最大化させる。
> 具体的な個人情報が少ないほど、「自分に似た誰か」を投影しやすい。

つまり、**情報を減らすことが、感情を増やす。**

## 5つの戒律

| # | 原則 | 具体的禁止 | 根拠 |
|:---:|---|---|---|
| 1 | **一人で完結する** | フォロー/フレンド/チャット/DM/コメント欄を作らない | 義務感を排除し、内発的動機のみで動く |
| 2 | **存在は感じる** | 匿名の集計データ(人数・割合・密度)のみ表示する | 社会的促進を最小コストで発動 |
| 3 | **自分の位置を知る** | 全体の中で自分がどこにいるかを%やチャートで示す | 競争ではなく座標。地図の中の●印 |
| 4 | **名前は要らない** | ユーザー名ではなく「47人」「上位3%」で語る | 情報を減らし、想像力を増やす |
| 5 | **押しつけない** | 「シェアしよう！」「友達を招待！」は一切ない | 自発的な行動だけが、本物のつながりになる |

> **これらに反する機能は、どれだけ便利でも、どれだけ競合が実装していても、作らない。**

## 位置・時間の匿名化ポリシー — 思想を守りながら安全を守る

> **最大のリスク: 「匿名で安心」のつもりが、位置・時間・希少性の組み合わせで個人が推測できる。**

### 3層プライバシーモデル

| レイヤー | 見える人 | 精度 | 例 |
|---|---|---|---|
| **Private** | 本人のみ | 正確 | 自分の観察一覧、正確な座標、時刻 |
| **Ambient** | 全ユーザー | **丸めた集計** | グリッド単位の密度、時間帯のみ |
| **Admin** | 管理者 | 正確（権限管理） | モデレーション、データ品質確認 |

思想を守りながら運用が破綻しない。「誰がいたか」ではなく「世界が動いてる」を見せるのに精度は要らない。

### 位置の丸め（グリッドラウンディング）

Ambientレイヤーで他ユーザーに見せる位置は、常にグリッドに丸める:

| シーン | 丸め精度 | 理由 |
|---|---|---|
| ストランドマップ | 500mグリッド | 密度の傾向が分かれば十分 |
| ゴースト | 1-2kmグリッド | 「このあたり」が分かれば十分 |
| Fog of War | 1kmグリッド | セル単位の探索済/未探索 |
| 完成度バー | エリア単位 | 地域全体の集計 |

**原則: 自分の観察は正確に見える。他人の観察は丸めて見える。**

### 時間の遅延

リアルタイムで他人の位置を見せると、「今あこにいる」が抜ける。

| 装置 | 最小遅延 | 表現 |
|---|---|---|
| ゴースト | **3時間** | 「今日、このあたりで」（「今」ではない） |
| ストランドマップ | **日次更新** | 昨日までの累積 |
| タイムカプセル | **月次** | 前年同月の集計 |

「数時間前に誰かがいた」と「今誰かがいる」は、体験としてはほぼ同じ。でも安全性は大きく違う。

### 保護種・希少種の強制ルール

> **これはcitizen scienceの必須装備。後付けは炎上する。最初から入れる。**

| カテゴリ | 処理 | 理由 |
|---|---|---|
| 環境省レッドリスト種 | 位置を**市町村レベル**に強制ぼかし | 密猟・採集リスク |
| 都道府県レッドリスト種 | 位置を**2kmグリッド**に丸め | 地域的な希少性 |
| 採集圧が高い種（昆虫等） | フラグで個別制御可能に | カブトムシ等 |

**実装**: `RedListManager.php` のカテゴリを利用。
観察保存時に種名をチェックし、該当すればAmbientレイヤーの座標を自動で丸める。
本人のPrivateレイヤーでは正確な座標が見える。

### ゴーストの過疎エリア対応（修正）

N=1でも発火するが、表示は丸める:

```
  ✕ 「他に1人が観察しています」
  ○ 「近くに気配があります」
```

人数を出さない。「気配」だけ。これでN=1でも個人特定のリスクが下がる。
N≥10のエリアでは「他に数人が観察しています」と曖昧表現。

## 技術制約 — 6つ目の戒律: システムに負担をかけない

> **原則: 全装置は PHP + SQL/JSON + 静的テンプレート だけで動く。AI APIは使わない。**

Ambient Presenceは「軽い」からこそ「ambient（環境的）」。
重い処理やAPI呼出があった瞬間、それは「ambient」ではなく「loading...」になる。

### 禁止事項

| 技術 | 禁止理由 |
|---|---|
| AI API (GPT/Gemini等) | コスト線形増。レスポンス不安定。スケールしない |
| WebSocket / リアルタイム通信 | RS Planは共用サーバー。常時接続は不可能 |
| 外部API依存 (リクエスト時) | レイテンシ追加。外部障害でikimonが止まる |
| 重いJSフレームワーク | フィールドで使うのに3G回線で3秒ロードは論外 |

### 各装置の技術構成

| 装置 | データソース | 処理方式 | キャッシュ |
|---|---|---|---|
| 🌲 ストランドマップ | observations.json の lat/lng | PHPで座標抽出 → JS描画 | 1時間キャッシュ |
| 🌿 完成度バー | COUNT(DISTINCT species) の成長推移 | PHP集計 | 日次更新 |
| 🔀 デトロイトチャート | wizard_choices.json | PHP集計 → %算出 | 同定完了時に更新 |
| 📊 タイムライン | observations の created_at | GROUP BY month → PHP | 日次更新 |
| 👻 ゴースト | WHERE created_at > NOW()-24h AND distance < 1km | PHPクエリ | リクエスト時（軽量） |
| 📖 ジャーニーマップ | ユーザー別observations | PHPで時系列ソート | ユーザーアクセス時 |
| � 同定エビデンス | id_evidences.json | PHPフォーム + JSON保存 | 同定時に更新 |
| 🌫️ Fog of War | 観察ゼロのグリッドセル | MapLibre fill-layer | 日次更新 |
| 🔊 環境音 | 近傍24hのobservationカウント | JS audioタグ | リクエスト時 |
| 🕰️ タイムカプセル | 前年同月のobservation座標 | PHP抽出 → JSオーバーレイ | 日次更新 |
| �📋 個人レポート | 上記の組み合わせ | テンプレート合成 | 期間終了時に1回生成 |

### キャッシュ戦略 — 事前計算で軽量化

```
  リクエスト時に毎回集計するもの:
    ・ゴースト（24h以内の近接観察。データ量が少ないので軽い）
    ・個人の観察カウント（WHERE user_id = ? で即返せる）

  日次cronで事前計算するもの（data/cache/に保存）:
    ・エリア別種数カウント → completion_cache.json
    ・月別観察数分布     → timeline_cache.json
    ・ストランドマップ用座標 → heatmap_cache.json
    
  イベント駆動で更新するもの（投稿/同定の時だけ）:
    ・同定統計           → id_stats_cache.json
    ・wizard選択統計     → wizard_stats_cache.json
```

**計算コストの目安（RS Plan環境）**:
- 観察1万件のCOUNT/GROUP BY: < 50ms
- JSONファイル読込（1MB以下）: < 10ms
- テンプレート合成: < 5ms
- **合計: 1リクエストあたり < 100ms。AI API不要。**

## 心理学的基盤 — なぜこの設計が「効く」のか

### 理論1: 自己決定理論（SDT）— Deci & Ryan, 1985, 2000

人間の内発的動機には3つの基本欲求がある:

| 欲求 | 定義 | Ambient Presenceでの充足 |
|---|---|---|
| **自律性** (Autonomy) | 自分の行動を自分で選んでいる感覚 | フォロー/コメントの義務なし。やるもやめるも自由 |
| **有能感** (Competence) | 「できた」「成長した」という手応え | 地域完成度バー、タイムラインでの位置、同定ルート |
| **関係性** (Relatedness) | 他者とつながっている感覚 | 匿名の足跡、選択統計、ゴースト |

**重要**: SNS型は関係性を強化するが、**自律性を犠牲にする**（返信義務、フォロー返し）。
Ambient Presenceは3欲求を**同時に満たす**。これがSDTが予測する最も持続的な動機づけ。

---

### 理論2: 過正当化効果 — Lepper, Greene & Nisbett, 1973

> もともと好きでやっていた活動に外的報酬（バッジ、ポイント、ランキング）を与えると、
> 報酬がなくなった瞬間に活動をやめる。
> **外的報酬は内発的動機を殺す。**

これが「実績バッジ」「ゲーミフィケーション」の最大の罠。
安易にバッジを配ると、**バッジのために投稿する人**が増え、バッジが止まった瞬間に離脱する。

**ikimonの対策**: 
- 実績バッジは**作らない**
- 「あなたは上位3%」は報酬ではなく**座標情報**（= informational feedback）
- Deci (1975) の認知評価理論: informational feedback（情報的フィードバック）は内発的動機を**促進**する。controlling feedback（統制的フィードバック）は**破壊**する

| 方式 | 種類 | 効果 |
|---|---|---|
| 「100回投稿で金バッジ！」 | controlling | 動機を破壊する |
| 「あなたは上位3%にいます」 | informational | 動機を促進する |
| 「34%の人が同じ道を選びました」 | informational | 動機を促進する |

---

### 理論3: 社会的促進 — Zajonc, 1965

> 他者の存在を**認識するだけ**で、単純課題のパフォーマンスが向上する。
> 相手が見知らぬ人でも、匿名でも、画面越しでも効果がある。

ストランドマップの匿名ピン、ゴーストの「今日2人が観察」はこれ。
**人の名前も顔も不要。「いる」と知るだけで人は動く。**

---

### 理論4: 最適弁別性理論 — Brewer, 1991

> 人は2つの矛盾する欲求を同時に持つ:
> 1. 集団に**所属したい**（同化欲求）
> 2. 集団から**際立ちたい**（差別化欲求）

デトロイトチャートの「34%が同じ道を選んだ」は両方を同時に満たす:
- 「34%も同じ」→ 自分は一人じゃない（同化）
- 「66%とは違う」→ 自分の選択はユニークだ（差別化）

**ランキング（順位型）は同化のみ、実績（バッジ型）は差別化のみ。**
デトロイトチャート型は**唯一、両方を同時に提供する。**

---

### 理論5: 心理的リアクタンス — Brehm, 1966

> 自分の自由が脅かされていると感じると、人はその行動に**反発**する。
> 「シェアしてね！」「お友達を招待！」は典型的なリアクタンス誘発。

Ambient Presenceが「押しつけない」のは好みの問題ではなく**心理学的必然**。
行動への圧力は、その行動の魅力を**減少させる**。

---

### 理論6: 社会的比較理論 — Festinger, 1954

> 人は自分を評価するために、他者と比較する。
> ただし上方比較（自分より上の人を見る）は動機を下げることがある。

| 比較の種類 | 例 | 効果 |
|---|---|---|
| 上方比較 | 「1位: ○○さん 5,000件」 | 「勝てない」→ やる気↓ |
| 下方比較 | 「あなたは○○さんより上」 | 一時的な満足 → すぐ飽きる |
| **水平比較** | 「同じくらいの人: 847人」 | 帰属意識 + 持続的に機能 |

ikimonのranking再設計は、意図的に**水平比較のみ**にする。

## やりすぎライン — アンチパターン集

> **バランスの原則**: すべての装置は「気づいたら嬉しい」であり「強制される」ものではない。

### ❌ 絶対にやらないこと

| パターン | なぜダメか | 理論 |
|---|---|---|
| 「○○さんが投稿しました！」通知 | 義務感を生む。見に行かないと罪悪感 | リアクタンス |
| 毎日ログインボーナス | 外的報酬。やめた瞬間に離脱 | 過正当化効果 |
| 「連続○日投稿！」ストリーク | 途切れた瞬間の喪失感が離脱を加速 | 損失回避バイアス |
| 「あと3件でバッジ獲得！」 | controllingフィードバック | 認知評価理論 |
| 他人の名前付きランキング | 上方比較で大半のユーザーが萎える | 社会的比較理論 |
| 「この投稿をシェアしよう！」ポップアップ | 自由への脅威 | リアクタンス |

### ⚠️ バランスが必要なこと（やりすぎ注意）

| パターン | OK | NG |
|---|---|---|
| 統計表示 | ページに1-2個、自然に埋め込む | 全ページに統計が溢れる |
| 「○人が〜」メッセージ | 画面の隅に小さく | モーダルで表示 |
| 地域完成度バー | 探索ページに1本 | 全ページにバーが出る |
| 「最初の発見者」表示 | 自分のタイムラインにのみ | 全員に見える形で表示（優越感の煽り） |
| ゴースト表示 | 数秒間のやわらかいパルス | 常にアニメーション |

### 📐 設計の判定テスト

新機能を追加する前に、この3問を通すこと:

1. **「これを無視しても、何も損しないか？」** → Yesなら実装OK。Noならリアクタンスを生む
2. **「これが毎回表示されても、うざくないか？」** → Noなら実装OK。Yesなら頻度を下げるか削除
3. **「これがなくても、サービスは使えるか？」** → Yesなら実装OK。Noなら強制になっている

### 装置5: 👻 ゴースト（同じ日・同じ場所の気配）

> 元ネタ: Journey — 名前のない旅の仲間が一瞬だけ現れる

地図上で、直近24時間以内に同じエリア（1km²グリッド）で
観察した人がいた場合、薄い光のパルスが一瞬だけ現れる。

```
  「今日、このあたりで他に2人が観察しています」
```

話せない。誰か分からない。
でも「今この瞬間、同じことをしている人がいる」。

これがJourneyの「名前のない旅の仲間」の翻訳。

**N=1でも発火する（過疎エリアの設計判断）**:

統計（装置3）はN<10で非表示にするが、ゴーストは**N=1でも表示する**。

理由: 過疎エリアで「自分以外に誰かがいた」は匿名性の問題ではなく**喜び**。
山の中で一人で観察している人にとって、「他に1人がこのあたりで観察しています」は
「仲間がいる」という発見。これを閾値で潰すのは本末転倒。

**仲間の増加通知（ゴースト拡張）**:

あるエリアで初めて別のユーザーが観察を投稿した時:
```
  「あなたが歩いたエリアに、新しい探索者が現れました」
```

これは通知に追加してよい（シナリオCの通知ルールに3つ目として追加）:
1. 同定完了通知
2. エリアの新種通知
3. **エリアの新規探索者通知** ← New

誰かは分からない。でも「自分のフィールドに誰かが来てくれた」。
これはDeath Strandingの「あなたのはしごが使われました👍」と同じ構造。

**実装は意外とシンプル**:
- 投稿のcreated_atとlat/lngから24h以内・半径1km以内を検索
- explore.phpの地図上に、淡いcircle markerを一時表示
- リアルタイムじゃなくていい。ページ読み込み時に判定


## 装置7: 🔬 同定エビデンスシステム（データ品質保証装置）

> **これは「コメント機能」ではない。科学的データの品質を保証する装置。**

### なぜ必要か

同定の精度がikimonのデータ価値を決める。しかし現在の設計には:
- 誤同定を検出する手段がない
- 同定の根拠を記録する場所がない
- 異なる意見を科学的に議論する仕組みがない

iNaturalistはコメント欄で補正している。だがコメント欄は「きれいですね！」も流入する。
**ikimonは「理由」しか入力できない構造にする。**

### 同定時の入力フロー

```
  同定: ナミアゲハ (Papilio xuthus)
  確信度: [Sure / Likely / Uncertain]  ← 必須

  同定の根拠（任意。ただし異議の場合は必須）:
  ┌─────────────────────────────────┐
  │  形態的特徴:                      │
  │  □ 翅の模様  □ 体色  □ 触角     │
  │  □ 体サイズ  □ その他            │
  │  補足: [                        ] │
  │  「後翅の赤斑が5つあり特徴的」     │
  │                                 │
  │  生態的証拠:                      │
  │  □ 食草  □ 行動  □ 生息環境     │
  │  □ 季節  □ 鳴き声               │
  │                                 │
  │  参考文献（任意）:                 │
  │  [学研の図鑑LIVE 昆虫 p.234     ] │
  └─────────────────────────────────┘
```

### 異議（Disagree）のフロー

別のユーザーが既存の同定に異議を唱える場合、**理由が必須**:

```
  異議: キアゲハ (Papilio machaon) ではないか
  確信度: Likely

  根拠（必須）:
  ┌─────────────────────────────────┐
  │  形態的特徴:                      │
  │  ☑ 翅の模様                      │
  │  「前翅基部が黒く塗りつぶされて    │
  │   いる。ナミアゲハは線状のはず」   │
  │                                 │
  │  生態的証拠:                      │
  │  ☑ 食草                          │
  │  「セリ科植物の近くで撮影。        │
  │   キアゲハの食草と一致」          │
  └─────────────────────────────────┘
```

### 設計判断

| 設計ポイント | 判断 | 理由 |
|---|---|---|
| 根拠は任意 or 必須？ | **同定時は任意、異議時は必須** | 気軽に同定できないと参入障壁が上がる。でも異議には根拠がないと荒れる |
| 自由テキスト？ | **選択式 + 補足テキスト** | 「きれい！」を構造的に排除。科学的カテゴリしか選択肢にない |
| 誰が書いたか見える？ | **見えない**（戒律4） | 理由の内容が重要。書いた人は関係ない |
| 反論の連鎖は？ | **2段階まで** | 同定 → 異議 → 再反論 で打ち止め。それ以上は管理者判断 |

### これが「ソーシャル機能の問い」への回答になる

| 求められる声 | 同定エビデンスでの充足 |
|---|---|
| 「お礼を言いたい」 | 同定が採用される = 最高のありがとう |
| 「議論したい」 | 異議 + 根拠で科学的対話が成立 |
| 「この人をフォローしたい」 | 不要。根拠の質が人の代わりになる |

**これはコメント機能じゃない。査読システム。**
学術論文のpeer reviewと同じ構造。感想ではなく証拠で対話する。

```
  実装: components/id_evidence_form.php [NEW]
  データ: id_evidences.json (同定ID + evidence構造体)
  表示: observation_detail.php に統合
```

### モデレーション設計 — 査読が炎上しないために

構造化だけでは攻撃的な文章やマウントは防げない。3層で守る:

**Layer 1: UIで押さえる**
- 補足テキストは**100文字以内**
- 入力欄のプレースホルダー: 「形態的特徴を具体的に記述」
- 感想（「きれい」「すごい」）は構造的に入力できない

**Layer 2: 信頼レベル（内部スコア、非表示）**

ランキングは禁止。でも内部スコアは持つ。見せないだけ。

| レベル | 条件 | できること |
|---|---|---|
| 新規 | 登録直後 | 同定のみ（理由任意） |
| 標準 | 同定実績5件以上 | 同定 + 理由付き同定 |
| 信頼 | 同定採用率80%以上かつ実績20件以上 | 同定 + 理由 + **異議提出** |

異議を出せるのは「信頼」レベル以上のみ。経験の浅いユーザーが専門家の同定に異議を唱えるリスクを排除。

**Layer 3: エスカレーション**
- 2段階（同定 → 異議 → 再反論）で打ち止め
- それでも決着しない場合: **管理者パネルが最終判断**
- 判断基準: 証拠の質、参考文献の有無、写真の特徴との整合性
- 判断結果は「管理者判断」タグで表示。誰が判断したかは非表示（戒律4）

## 📋 個人レポート — AIゼロで「良い感じ」を出す

> **原則: AIは使わない。テンプレート × データ = 感動。**

Spotify Wrappedを思い出してほしい。
あれはAI生成じゃない。再生データ + 固定テンプレート + 良いデザイン。
それだけで毎年何億人がスクショしてシェアする。

ikimonの個人レポートも同じ構造で作る。

### レポート生成の仕組み

```
  データ（MySQL/JSON）       テンプレート（PHP）
  ──────────────        ──────────────
  ・観察件数: 47          「あなたは{n}種の生き物と
  ・種数: 23              出会いました」
  ・エリア数: 5           
  ・最初の観察: 3/15      「最初の出会いは{date}の
  ・最多観察種: ナミアゲハ   {species}でした」
  ・同時期の観察者: 89人    
                         「同じ時期に観察した
         ↓               人は{n}人」
  ──────────────────────────
       テンプレート合成
  ──────────────────────────
         ↓
    完成品レポート（HTML/PDF）
     ※ AIコスト: ¥0
     ※ 生成速度: <1秒
```

### 3層レポート

**Layer 1: マイクロレポート（同定完了時）** — 自動表示

```
  誰かがあなたの発見に名前をつけました。

  ナミアゲハ (Papilio xuthus)
  この種を記録した人の中で、あなたは上位12%の早さでした。
```

→ テンプレートの変数: `{species_name}`, `{percentile}`

**Layer 2: 季節/期間レポート** — 期間終了時に自動生成

```
  ── あなたの2026年 春（3月〜5月）──

  出会った種: 23種
  歩いたエリア: 5箇所
  貢献した完成度: 静岡市葵区 +4.2%

  あなたの春の一枚:
  [最もいいねが多かった観察の写真]

  同じ春を歩いた人: 89人
```

→ すべてSQLクエリ + テンプレート。AI不要

**Layer 3: 年間レポート** — 年末に生成（Spotify Wrapped相当）

```
  ── あなたの2026年 ──

  総観察: 128件
  出会った種: 67種
  歩いた距離: 推定 48km
  最も活動した月: 5月（34件）
  最初の発見: 3月15日のナミアゲハ
  最もレアな発見: ハンミョウ（全体の0.3%しか記録していない種）
  
  あなたの種ポートフォリオ:
  🦋 チョウ目 ████████ 34%
  🪲 甲虫目  ████░░░░ 22%
  🐦 鳥類    ███░░░░░ 18%
  🌿 植物    ██░░░░░░ 15%
  🐸 両生類  █░░░░░░░  6%
  📦 その他  █░░░░░░░  5%

  ── 世界の中のあなた ──
  同じくらいの探索者: 234人
  あなたは上位 8% の多様性記録者です

  「2026年、あなたは67の命と出会いました」
```

### なぜAIを使わないのか

| 方式 | コスト | 速度 | 品質の安定性 | スケーラビリティ |
|---|---|---|---|---|
| AI生成テキスト | API呼出 ¥/回 | 2-5秒 | 不安定（毎回違う） | ユーザー増でコスト線形増 |
| **テンプレート合成** | **¥0** | **<1秒** | **完全に安定** | **ユーザー無限でもコスト変わらない** |

- テンプレートの文体は**一度だけ人間が丁寧に書く**。それを全ユーザーに使い回す
- データが変わるからレポートが個人的になる。文体を変える必要はない
- 「あなたは67の命と出会いました」— この一文にAIは要らない。**データが語る**

### テンプレートの設計原則

1. **主語は「あなた」**: 「ユーザーの総観察は〜」ではなく「あなたは〜と出会いました」
2. **数字は1つに絞る**: 1画面に数字は1-2個。多すぎると情報過多
3. **比較は水平**: 「上位8%」はOK。「1位の人は5,000件」はNG（上方比較）
4. **最後に意味を添える**: 「67の命と出会いました」のような一文で締める。事実だけど詩的になる

## この設計が「世界を制する」理由

1. **SNS疲れの受け皿**: 2026年、人々はSNSに疲れている。でも「完全に一人」も寂しい。ikimonはその中間を埋める
2. **科学データとの両立**: 匿名集計データはGBIF提出にも最適。プライバシーとデータ品質が共存する
3. **コールドスタートに強い**: ユーザーが5人でも50人でも統計は出る。「5人中あなたは3番目」でもAmbient Presenceは機能する
4. **文化を選ばない**: フォロー/コメント文化は文化圏で異なる。匿名の痕跡は万国共通
5. **持続可能**: SNSモデルは「投稿しないと存在が消える」。Ambient Presenceは「一度の足跡が永遠に地図に残る」

## 同定 = ikimon最上位のストランド

> **同定は「いいね」の上位互換ではない。次元が違う。**

| 行為 | 相手へのインパクト | 情報量 | 科学的価値 |
|---|---|---|---|
| いいね 👍 | 「見たよ」 | ゼロ | ゼロ |
| コメント 💬 | 「きれいですね」 | 低い | ゼロ |
| DM ✉️ | 個人的やりとり | 高いが閉じている | ゼロ |
| **同定 🔬** | **「これはナミアゲハだよ」** | **極めて高い** | **データの完成** |

同定は:
- 相手の写真を**注意深く見て**
- 自分の知識を**無償で提供して**
- 観察データの科学的価値を**完成させる**

**Death Strandingの「はしご」は、ikimonでは「同定」。**

誰かが崖の前で立ち止まっている（種名が分からない投稿）。
名前も知らない誰かが、はしごをかける（種名を同定する）。
置いた人は、相手が渡れたかどうか知らない。
でも**世界の地図は、少しだけ完成する。**

### 同定が他のインタラクションを全て包含する

```
  いいね: 「見たよ」
  → 同定: 「見た上で、何か分かったよ」（いいね + 知識）

  コメント: 「すごいですね」
  → 同定: 「これはアゲハチョウだよ」（感想ではなく事実）

  フォロー: 「あなたの投稿をもっと見たい」
  → 同定した人の投稿は、自然に目に入る（明示的フォロー不要）
```

### だから「いいね」機能は再定義する

現在の `toggle_like.php` は残す。ただしその意味が変わる:
- ❌「素敵な写真ですね」のいいね → SNS的。情報量ゼロ
- ✅ Death Strandingの「👍」 → **「ここにいてくれてありがとう」の無言の合図**

表示方法:
- push通知は**しない**（戒律5違反）
- 自分の投稿を見に行った時に、静かにカウントが見える
- 「3人がこの観察に👍しました」— 誰かは分からない

## 運用シミュレーション — 矛盾の検出と解決

### シナリオA: コールドスタート（総ユーザー3人）

> **問題**: 「34%が同じ道を選んだ」= 1人。匿名性が破綻する。

```
  ID Wizard完了:
  「あなたと同じルートを辿った人: 1人」
  → 「あ、あの人だな」と特定できてしまう
```

**解決**: 最小表示閾値を設ける

| N (総ユーザー) | 表示方法 |
|:---:|---|
| N < 10 | 割合・人数を**表示しない**。フローチャートのルートのみ表示 |
| 10 ≤ N < 50 | 「少数の人が同じ道を選びました」（曖昧表現） |
| N ≥ 50 | 「34%が同じ道を選びました」（正確な割合） |

地域完成度バーは**N=1でも矛盾しない**（種数は個人を特定しない）。
ストランドマップも**N=1でも機能する**（ピンは観察の位置であり個人の位置ではない）。

### シナリオB: プロの同定者が来た。どう気づくか？

> **問題**: 戒律5「押しつけない」があるのに、未同定の投稿をプロにどう知らせるか。

```
  プロは通知を受けない（戒律5）。
  でも未同定の投稿が溜まっている。
  誰もプロに「見てほしい」と言えない（戒律1）。
  → 永遠に未同定のまま？
```

**解決**: プロが**自分から見に行く場所**を用意する

- `id_center.php` = プロの同定作業台。**プッシュしない。プルする**
- 表示: 「未同定の観察: 48件（昆虫: 23件 / 植物: 12件 / 不明: 13件）」
- プロは好きな時に好きな分類群だけ見に行く
- **「仕事」ではなく「散歩」のメタファー**: 同定センターは「図書館の書架」。行きたい時に行く

さらに、同定完了後にプロが見る画面:
```
  今日の同定: 5件
  あなたの同定と一致した人: 89%
  「あなたはこのエリアで最も正確な同定者の一人です」（N≥50の場合のみ）
```

これは**controlling（統制的）ではなくinformational（情報的）**。プロの有能感（SDT: Competence）を満たす。

### シナリオC: 通知はどこまで許容するか

> **問題**: `toggle_like.php`と同定結果の通知。戒律5「押しつけない」との整合。

**判定基準**: 「あなたの行動に対する反応」は通知OK。「他人の行動への誘導」はNG。

| 通知内容 | 判定 | 理由 |
|---|---|---|
| 「あなたの観察が同定されました」 | ✅ | 自分のデータが完成した報告。科学的価値がある |
| 「あなたの観察に👍がつきました」 | ⚠️ → **表示しない** | 情報量ゼロ。自分で見に行った時に分かればいい |
| 「○○さんが新しい投稿をしました」 | ❌ | 他人の行動。リアクタンス誘発 |
| 「このエリアに新しい種が追加されました」 | ✅ | 共有世界の更新。個人ではなく世界の変化 |
| 「あと3件で〜」 | ❌ | controlling feedback |

**結論**: 通知は3種類だけ

1. **「あなたの観察が同定されました: ナミアゲハ」** — 同定が完了した報告
2. **「あなたのエリアに新しい種が記録されました」** — 世界の変化の報告（週1まとめ）
3. **「あなたが歩いたエリアに、新しい探索者が現れました」** — 仲間の発見

それ以外の通知は作らない。

## 移行パス — 現状からAmbient Presenceへ

現在のikimonには既存の機能がある。一気に切り替えると既存ユーザーが混乱する。
段階的に移行する。

| Phase | やること | ユーザーから見た変化 |
|---|---|---|
| 1. **足す** | 完成度バー、ストランドマップ、ゴーストを新規追加 | 「新しい機能が増えた」（ポジティブ） |
| 2. **慣らす** | 新UIが定着するまで旧機能と並存 | 両方使える安心感 |
| 3. **再定義** | ranking.phpを「探索者の座標」に、いいねをDS型👍に | 「意味が変わった」が自然に感じる |
| 4. **削る** | 不要になった旧UI要素を静かに削除 | 気づかないくらい自然 |

**一番重要なルール: 「機能が消えた」と感じさせてはいけない。「良くなった」と感じさせる。**

## 測定方法 — Ambient Presenceの成否をどう知るか

> **これはKPIでは測れない。ユーザーに聴くしかない。**

「なんか良い」は主観。ダッシュボードの数字には出ない。
他のSNSの比較データもない。前例のない設計だから。

**測定手段**:
- 実際に使っているユーザーへの**継続的なヒアリング**
- 「この機能、どう感じた？」と直接話を聞く
- 観察イベントやワークショップでの対面フィードバック
- 季節ごとのユーザーインタビュー（3-5人）

**測れないものを測らない。** データで測れるのは行動（投稿数、リテンション）。
Ambient Presenceの成否は「従る」かどうか。それは人の言葉でしか分からない。

## 🛠️ 未決事項 — 実装しながら継続検討

### ソーシャル機能の解決策 → 同定エビデンスシステムで実質解決

**元の問題**: 「同定してくれた人にお礼言いたい」「フォロー機能がほしい」の声が必ず来る。

**解決**: 装置7「同定エビデンスシステム」がこれに答える。
- 「お礼を言いたい」→ 同定採用が最高のお礼
- 「議論したい」→ 異議 + 根拠で科学的対話が成立
- 「フォローしたい」→ 不要。根拠の質が人の代わり

ただし、これがユーザーにとって十分かは**実装後に検証**する。
「やっぱりお礼が言いたい」という声が続くなら、戒律に死角がない形で追加措置を検討。
全戒律に共通する精神: **求められる前に作るな。求められた時に考えろ。**
