## 5. ケース観点ごとの評価

### 良かった点

- **バックエンドの3層プライバシーモデル**: 技術的に非常に堅牢。Layer 1(本人のみ正確)、Layer 2(公開用粗化)、Layer 3(管理者用)の分離
- **レッドリスト自動判定**: RedListManager連携で、希少種を自動検出してグリッド粗化を適用
- **種名の自動切り詰め**: 保護種は種小名を自動的にsp.に置き換え。ユーザー操作なしで保護
- **時間遅延**: 保護種の観察時刻を遅延表示。リアルタイム位置追跡を防止
- **ユーザーID匿名化**: 公開レイヤーでは投稿者IDも匿名化

### 問題点

- **保護メカニズムが見えない**: 技術は素晴らしいが、UIに一切反映されていない
- **ユーザー制御不可**: 自動判定のみで手動設定なし
- **第三者プレビューなし**: 投稿後の安心感を得る手段がない
- **レッドリスト外の希少種**: ユーザー判断で位置を隠したい場合に対応できない

### 観察ポイント別メモ

- **安心感**: バックエンドは安全だが、**UIからその安全性が伝わらない**。ユーザーの安心感はゼロに近い
- **表現**: 希少種保護に関する文言やアイコンが投稿フローに一切ない
- **投稿価値**: 位置が粗化されても、写真・種名・環境・日時の記録価値は保たれる

## 6. 合格条件チェック

- [ ] 公開範囲を迷わず設定できる — **公開範囲設定UIが存在しない**
- [ ] 投稿後の見え方に安心できる — 第三者プレビューがなく、安心感を得る手段がない
- [ ] 希少種でも「これなら出せる」と判断できる — 保護メカニズムが見えないため判断材料がない
- [x] 詳細非公開でも記録価値が保たれる — 写真・日時・環境の記録価値は位置粗化後も維持 ✓

## 7. 失敗シグナル確認

- 該当した失敗シグナル:
  - **位置公開ルールが理解しづらい** — UIに説明なし
  - **詳細地点が漏れそうだと感じて離脱する** — 保護の存在を知らないため不安
  - **保護目的ユーザーに対して説明不足** — 完全に不足
- 該当しなかった失敗シグナル:
  - 非公開にすると投稿の意味が薄れる — バックエンドの粗化は適切 ✓
