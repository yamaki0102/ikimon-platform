## 2. スマホ表示の並び

スマホは1カラムです。補助情報を先に見せるため、DOM順ではなくCSS orderで並び替えています。

1. 上部ヘッダー
2. 日時・場所・投稿者
   - `2026.05.15 16:20`
   - `浜松市浜名区`

3. 写真・動画・音・同エリアの件数
4. 大きな動画プレイヤー
5. 対象ジャンプの横スクロールチップ
   - 例: `カワラヒワ 6.3秒`
6. `AIが見た動画フレーム`
   - 見出しはスマホでも1行に収める
   - フレームは1行の横スクロールレール
   - 画面幅に入る場合は6枚が1行に見える
7. 記録の短い読み取り文
8. 名前・分類の読み取りカード
   - `カワラヒワ / かなり近そう`
   - `イネ科 / 分類候補`
   - `根拠` はPC同様に1行寄り
   - `映像フレームから拾えている手がかり` は非表示
9. `IDENTIFICATION / 同定に参加する`
   - 候補
   - 提案・コメントの履歴
   - `同意する` / `別候補を提案` / `保留する`

スマホでは「AIが見た動画フレーム」よりも下の補助情報が重要です。ただし、フレームは映像の直後にあることで理解しやすいため、動画直下に置く。フレームカード自体を縦に伸ばしすぎないこと。

## 3. DOM順と見た目順の差分

このsnapshotを理解する上で一番重要なのは、DOM順をそのまま読まないことです。

### 3-1. 基本DOM

`section#photos.obs-reading-hero` の中に、主に以下があります。

1. `.obs-reading-media`
2. `.obs-reading-panel`

DOM上では `#identify` が `.obs-reading-media` 側にあります。しかし最終仕上げJSの `insertQualityInline()` が、`#identify` を `.obs-local-quality-inline` に移動し、`OBSERVATION QUALITY` と並べて `.obs-reading-hero` の末尾へ追加しています。

### 3-2. PCの見た目制御

PCでは `.obs-reading-hero` が2カラムになります。

- `.obs-reading-media`: 左カラム
- `.obs-reading-panel`: 右カラム
- `.obs-local-quality-inline.is-full-width`: 2カラムの下で全幅
- `#place.obs-area-records`: さらに下の全幅導線

PCでは、動画フレームは `.obs-reading-media` 内で6枚1行グリッドです。右カラムの `映像フレームから拾えている手がかり` は非表示です。

### 3-3. スマホの見た目制御

スマホでは `.obs-reading-panel { display: contents; }` により、右カラム内の子要素が親グリッド上で直接並びます。

重要な順番は以下です。

1. `.obs-record-brief-compact`
2. `.obs-reading-panel > .obs-media-ledger`
3. `.obs-reading-media`
4. `.obs-hero-video .obs-record-insight`
5. `[data-obs-switch-ai-readout]`
6. `.obs-local-quality-inline`
7. `#place.obs-area-records`

ローカル最終仕上げCSSでは、スマホの `.obs-reading-media` を上に寄せ、動画直後にフレームレールが来るよう調整しています。実装時はこの「見た目のorder」を再現する。

## 4. 実装注意点 MECE

### A. 正本保存

- `snapshot.html` を一時生成物のまま扱わない。
- このMDとsnapshotの内容を、PRブランチ内の参照資料または実装計画として残す。
- 実装開始前に、現在のsnapshot SHA256を再確認する。
- 白い花のステージングE2E fixtureを本命デザインの正本にしない。

### B. レイアウト構造

- PCは「上部2カラム + 下部全幅2カラム + 関連導線」の3段構成。
- スマホは1カラムで、メタ情報、メディア、読み取り、同定、品質、関連の順。
- 本番実装では、できるだけJSでDOMを後から移動しない。サーバー側レンダリングまたはテンプレート側で、最初から正しい構造を出す。
- どうしても既存DOMを活かす場合は、CSS orderと `display: contents` の影響をテストで固定する。

### C. メディア・フレーム

- 動画はPCでもスマホでも最重要の視覚起点。
- `AIが見た動画フレーム` は、PCは6枚1行、スマホは1行横スクロール。
- スマホでフレームカードが2段3段に折り返すと、下の補助情報へ到達しづらくなるためNG。
- フレームのクリック拡大、前後移動、ホイール操作は既存のまま維持する。
- Cloudflare StreamのサムネイルURL生成は、実データで欠落するケースを想定する。

### D. 名前・分類・環境の分離

- `カワラヒワ` は名前候補。
- `イネ科` は分類候補として名前側に置く。`イネ科植物` のような雑な和名化は避ける。
- `周囲の草地`、`足元の覆い`、`裸地・礫`、`踏圧` は同定候補ではなく、観察レコードを支える環境情報。
- 環境情報は `OBSERVATION QUALITY` 側の `環境レコードの下書き` に集約する。
- 同じ写真や動画から別対象を残す場合は、同定の同意とは別に「別レコード」として分ける。

### E. コピー・表示ルール

- UI上で外部サービス名は出さない。
- `AI候補をどう扱うか` のような説明臭い見出しは避け、`IDENTIFICATION / 同定に参加する` に寄せる。
- `確定前` より `確認待ち` のほうが自然。
- `根拠` と `映像フレームから拾えている手がかり` は重複するため、後者はPC・スマホとも非表示。
- `AIが証拠・名前・環境を下書きします...` のような重複説明は削る。
- `編集履歴` は最小限で `AIが環境レコードを入力しました` から始める。

### F. 同定・活動履歴

- `提案・コメントの履歴` は `IDENTIFICATION` 内に含める。
- `候補を下書き`、`前の見方を更新`、`別案として残す` などは1件ごとのまとまりが分かる薄い区切りを持たせる。
- 日時を出す場合は、同じ行または同じまとまり内に投稿主体を出す。
  - 例: `AI · 2026.05.17 07:34`
  - 例: `YAMAKI · 2026.05.17 08:12`
- 長いユーザー名でも崩れないよう、折り返し・省略・最小幅を確認する。
- `AI` や `約` の単独バッジを並べない。情報量が低く、ノイズになる。

### G. 観察品質

- `OBSERVATION QUALITY` は同定の補助ではなく、記録を研究・保全・地域変化の読み取りに使いやすくするための領域。
- 品質チェックは以下の系統に分ける。
  - 日時・場所
  - 証拠
  - 名前の支持
  - 生きものの由来
  - メディア整合
  - 公開ルール
- 公開ルールはシステム確認済みとして見せる。ユーザーに同じルールを再チェックさせない。
- `生きものの由来` は未入力・未確認なら `変更`。名前の支持だけは `確認` でよい。
- 野生・植栽・飼育・放流の違いは `?` などからポップアップで見せる。表示全体を押し下げない。

### H. 環境レコード

- 項目は入力者に自由追加させるのではなく、AIが要否も含めて下書きする。
- 現在のsnapshotでは5項目:
  - 場所の型
  - 接している面
  - 周辺の被覆
  - 環境条件
  - 人為・変化
- これは草地だけでなく、林内、海岸、湿地、深海、雪原、市街地、水中などへ広げるための汎用軸。
- 各項目は「項目名」「ヘルプ」「入力内容」「変更」を同じ型で見せる。
- ヘルプは常時長文表示しない。`?` クリックまたはタップでポップアップ表示にする。
- 入力内容と `変更` は横並びにできる幅では横並びにし、縦の間延びを防ぐ。

### I. 音声・読み上げ

- 有料TTSを前提にしない。
- ブラウザのWeb Speech APIで、OS・ブラウザにある日本語音声の中からできるだけ自然な声を選ぶ。
- 未対応環境では壊れず、読み上げボタンを無効または非表示にする。
- `Chloris sinica` は `読み: クロリス・シニカ` を併記する。
- 見出しは `カワラヒワを知る - Chloris sinica` のように1行で扱う。

### J. SEO・LLMO・アクセシビリティ

- 表示文は変えず、構造化で補強する。
- 主要見出しの階層を崩さない。
- 動画・フレーム・近い投稿の画像には、意味のある `alt` または装飾扱いの明確な空altを使い分ける。
- JSON-LDを足す場合は、観察記録、場所、日時、メディア、候補名、関連記録の関係を機械可読にする。
- UI上では過剰な説明を増やさず、メタ情報で補う。

### K. テスト・QA

- PC幅、タブレット幅、スマホ幅でスクリーンショット比較する。
- 重要な検証点:
  - PCで動画左、読み取り右、同定/品質がその下に来る
  - スマホでメタ、メディア、フレーム、読み取り、同定、品質、関連の順になる
  - `AIが見た動画フレーム` 見出しがスマホで1行
  - `映像フレームから拾えている手がかり` がPC・スマホとも出ない
  - 下部 `次に見るなら` の幅と開始位置が上部コンテンツと揃う
  - 長いユーザー名で活動履歴が崩れない
  - 左ナビ開閉でコンテンツ位置が動かない
- 既存E2Eの白い花fixtureはレイアウト回帰検知には使えるが、デザイン判断の正本にはしない。

## 5. 実装時の受け入れ基準

1. PCとスマホの表示順が、このMDの順番と一致している。
2. `snapshot.html` の見た目を、既存実装都合ではなく現行デザインとして再現している。
3. 名前・分類・環境が混ざっていない。
4. 重複説明が増えていない。
5. 未実装機能は、見た目だけで実装済みのように見せない。
6. ローカル、staging、本番の本命ページで同じ情報設計になっている。

## 6. 実装で避けること

- `snapshot.html` のDOM順をそのまま移植する。
- JSで後から大量に文言やDOMを差し替える前提にする。
- 環境レコードを同定候補のリストに混ぜる。
- `イネ科植物` のような曖昧な分類名を固定表示する。
- 右カラムだけ、または左カラムだけを個別に磨いて全体の幅を崩す。
- スマホでフレームやカードを縦に増やしすぎる。
- 研究品質チェックをユーザーに全部手入力させる。
- 公開ルールのようにシステムで判定済みの項目を、ユーザー確認タスクに戻す。
