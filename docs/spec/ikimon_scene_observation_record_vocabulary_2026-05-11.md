# ikimon.life Scene / Observation Record Vocabulary

更新日: 2026-05-11

## 結論

ikimon.life のユーザー体験では、写真・動画・音声・場所・時刻・気づきを束ねる入口を **シーン** と呼ぶ。
国際連携や研究利用の単位は **観察レコード** と呼び、これは Darwin Core / GBIF の `Occurrence` に対応する。

## 用語の役割

| 表示語 | 内部/標準の対応 | 役割 |
|---|---|---|
| シーン | visit / event / media bundle | ユーザーが残す場面。写真・動画・音声・場所・時刻・メモを束ねるハブ |
| 観察レコード | occurrence / Occurrence | シーン内の対象ごとの記録。現在採用する同定を 1 つ持つ |
| 記録対象 | subject / candidate | AI または人がシーン内で見つけた対象。採用されると観察レコードになる |
| メディア | media / evidence asset | 写真・動画・音声。複数の観察レコードの証拠になり得る |
| 同定 | identification | 観察レコードに対する名前付け。履歴を持てる |

## 設計ルール

1. 1 シーンから複数の観察レコードを作れる。
2. 1 観察レコードの現在同定は 1 つを基本にする。
3. 同定候補、投稿者同定、専門家修正は履歴として保持する。
4. AI が見つけた対象は、ユーザーが採用するまで観察レコードとして扱わない。
5. 背景に写った小さな対象をすべて自動レコード化しない。ノイズを防ぐため、確度・有用性・ユーザー意思を条件にする。
6. 公開面では「投稿」を主語にしない。入口は「シーンを残す」、一覧は「シーンライブラリ」、標準データは「観察レコード」と呼ぶ。

## UI 方針

- `/record`: シーンを残す場所。
- `/notes`: シーンライブラリ。写真・動画・音声・ガイド成果を見返し、対象ごとの観察レコードへ育てる場所。
- `/observations`: 観察レコード一覧。公開され、同定・証拠・対象単位で確認する場所。
- `/scenes/{id}/targets`: シーン内の **記録対象** を整理する場所。画像の範囲、動画・音声の時間区間、対象ラベル、メモ、AI候補への判断を扱う。
- 観察詳細: シーンの証拠を見せつつ、対象ごとの観察レコード・同定履歴・AI 候補を切り替えられる場所。

## SceneTarget 初期実装

`SceneTarget` は、シーンから切り出された記録対象を、観察レコード化の前段として保持する。

- 保存先: `scene_targets`
- 主キー: `scene_target_id`
- シーン参照: `visit_id`
- メディア参照: `asset_id`
- AI候補参照: `candidate_id`
- 将来の観察レコード参照: `occurrence_id`
- 画像範囲: `normalized_rect`
- 動画・音声区間: `start_second`, `end_second`
- 判断状態: `draft`, `adopted`, `ignored`, `later`, `converted`

AI候補を `adopted` にしても、ただちに確定同定済みにはしない。初期実装では `SceneTarget` として採用し、観察レコードへの変換は後続ワークフローに委ねる。

## 投稿直後フィードバック

保存直後の主語は同定ではなく **シーンの解像度** にする。

- 良い: 場所、時間、周囲、音、動き、再訪比較、見なかった状況が残ったことを返す。
- 良い: 「同じ場所で比べる解像度が上がった」「次に見返す手がかりが残った」と返す。
- 悪い: 「種に近づいた」「属まで分かった」「同定ポイントが増えた」を投稿成功の主報酬にする。
- 同定の案内は観察レコード詳細・AI 候補・同定履歴のレイヤーで扱う。

## 禁止する混線

- メディア 1 個 = 同定 1 個、と説明しない。
- シーンと観察レコードを同じ意味で使わない。
- 「ノート」を主要概念にしない。音声・動画・複数対象の感覚が弱くなるため、読み物的な比喩に限定する。
- 「投稿」を主要 CTA にしない。SNS 投稿感が強く、研究利用・再訪・同定履歴の文脈を弱める。
