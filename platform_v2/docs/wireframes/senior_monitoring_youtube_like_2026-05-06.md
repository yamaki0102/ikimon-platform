# Senior Monitoring UI Wireframes 2026-05-06

目的: ステージング反映前に、年配/熟練モニターが多い前提でトップページと観察投稿一覧の大幅UI刷新案を比較する。

対象:

- トップページ
- 観察投稿一覧
- 説明/読み物系の退避先

参照:

- `platform_v2/docs/ikimon_monitoring_current_state_and_requirements_2026-05-05.md`
- `docs/review/shizuoka-ux-tests/SZ-UC-05_report_codex.md`
- YouTube のホーム型コンテンツ棚
- iNaturalist の Observations 画面: filters / summary stats / Map-Grid-List

## 1. 判断の前提

この刷新の主成果は「きれいなトップ」ではなく、次の4つ。

| 成果 | 画面上の意味 |
|---|---|
| 3秒理解 | ここで何をするかが説明文なしで分かる |
| 低負荷参加 | 投稿、同定、地図、マイページへ大きな入口で到達できる |
| 継続観察 | 再訪問、定点、レビュー待ち、地域ミッションが見える |
| 品質向上 | `review_ready` / `monitoring_ready` を利用者にやさしい言葉で返す |

年齢バイアスについては、対象MDの `4.7 参加者属性と年齢バイアス` に整理済み。BioMonWeek公式に年齢構成データは見当たらないが、生物多様性記録者・市民科学・モニタリングサイト1000周辺では中高年/熟練者寄りの根拠がある。設計上は「年配者だけに寄せる」ではなく、熟練者の地域知と継続力を殺さず、若年/学校層が学習して役割を増やせる導線にする。

## 2. 情報設計

トップでは説明を削り、行動とコンテンツ棚を前面に出す。

| 置く | 置かない/退避する |
|---|---|
| 観察する、同定する、地図を見る、マイページ | 長い理念文、制度説明、BioMonWeek解説 |
| 今日の地域フィード | 研究/企業向け詳細 |
| 同定待ち、再訪問候補、近くの記録 | Darwin Core / export / governance 詳細 |
| 読み物への1本導線 | 複数の説明セクション |

読み物系は `/learn` または新設の「知る」サブページへ退避する。トップでは最大1段だけ「はじめての方へ」を出す。

## 3. トップページ案

### 案A: YouTube Home Feed + Local Map

主案。トップの第1ビューを、検索/大CTA/横スクロール棚に寄せる。加えて、案Cの地域マップを「地図ファースト」ではなく、トップ内の主要コンテンツ棚として取り込む。

- 大ボタン: `観察する`, `同定する`, `地図`, `マイページ`
- 棚: `近くの観察`, `同定待ち`, `地域マップ`, `読み物`
- 長所: 最も分かりやすい。説明を読ませず、目的行動へ進める。
- 短所: モニタリング文脈を弱く見せすぎるリスク。ただし地域マップ棚で補える。
- 採用条件: 初回訪問者/年配者が迷わず4主要機能へ入れる。

補足: `再訪問しよう` と直接促しても、多くの参加者は動かない前提で設計する。再訪問は主CTAにせず、`前にも記録がある場所`, `この季節に増えた記録`, `近くで変化が見える場所` として地図/地域棚の中に埋め込む。

### 案B: Task Hub + Feed

最初に「今日やること」を置き、その下にコンテンツ棚を置く。

- 主要タスク: `写真で記録`, `名前を確かめる`, `同じ場所を再訪問`
- 補助: 今日の進捗、未完了、地域の依頼
- 長所: モニタリング参加者の継続行動に強い。
- 短所: YouTubeらしさは少し弱い。
- 採用条件: 継続利用者の再訪問率、同定参加率を主KPIにする場合。

### 案C: Map First + Local Feed

改善済みのマップをトップの主役にする。地域モニター/自治体/企業向けに強い。ただし今回の主案では、トップ全体を地図ファーストにせず、案Aの中の地域マップ棚として採用する。

- 上部: 大きな地域マップ
- 下部: 周辺の観察、未同定、再訪問候補
- 長所: 「地域の自然を見ている」感が強い。
- 短所: スマホ初回では地図操作が重い可能性。
- 採用条件: マップの表示速度、誤タップ、現在地許可なし状態が問題ない。

## 4. 観察投稿一覧案

### 案1: Video Grid Style

YouTube/iNaturalistの中間。写真サムネイルを大きくし、カード内テキストは最小化。

- 表示: 写真、名前候補、場所丸め、状態ラベル
- フィルタ: `すべて`, `同定待ち`, `近く`, `写真あり`, `地域マップ`
- 長所: 年配者にも見やすい。カード単位で判断しやすい。
- 短所: 一覧密度は低い。

### 案2: Review Queue Style

熟練/レビュアー向け。状態、証拠、要対応を大きく出す。

- 表示: `同定待ち`, `追加写真が必要`, `専門家確認`, `公開粒度確認`
- 長所: 熟練者の同定/レビュー参加に強い。
- 短所: 一般投稿者には業務画面に見えやすい。

### 案3: Map + List Split

iNaturalist Observations の Map/Grid/List 発想を採用し、地図とカードを並べる。

- 表示: 左にリスト、右に地図。スマホでは地図/一覧を切替。
- 長所: 地域性と観察一覧を同時に見られる。
- 短所: 実装と性能チェックが重い。

## 5. 推奨

ステージング前の次フェーズでは、トップは **案A: YouTube Home Feed + Local Map**、観察一覧は **案1: Video Grid Style** を主案にする。

理由:

- 今回の課題は「文字が多くてきつい」なので、まず読む量と判断点を減らすべき。
- 年配/熟練者には、カード・大ボタン・状態ラベルのほうが負荷が低い。
- モニタリング品質の要素は、説明文ではなく `同定待ち`, `地域マップ`, `確認済み`, `追加写真` の行動棚に埋め込める。
- `再訪問` は押し出しすぎない。直接の依頼ではなく、地域マップ内の変化・前回記録・季節差分として見せる。
- 案B/Cと観察一覧案2/3は、ログイン後・熟練者モード・地域管理者モードとして後から足せる。

## 6. 実装時の品質ゲート

- 主要ボタンの高さは56px以上。
- カード内テキストは見出し1行、補足1行、状態ラベル1つを基本にする。
- トップの説明文は1画面目に置かない。置く場合も1文だけ。
- `AI候補` と `確認済み` を混ぜない。
- `確定` は専門家/信頼レビュアー確認以外で使わない。
- 地図/フィード/同定待ちの切替で横スクロールを出さない。
- 低視力想定で、色だけに依存せずラベルを併記する。
