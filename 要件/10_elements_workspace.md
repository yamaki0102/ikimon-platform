## 心理学的設計原則

ikimonの全設計判断は以下の理論に基づく。「なぜこのUIにするのか？」の根拠として参照。

| 理論 | 核心 | ikimonでの体現 | 実装ファイル | 実装度 |
|---|---|---|---|:---:|
| **自己効力感** (Bandura) | 「私にもできる」の信念 | 3タップ投稿で「できた！」→ ガイド付き同定で「分かった！」→ 成功体験の積み上げ | `post.php` (Confetti演出), `id_wizard.php` | 🟡 |
| **集団効力感** | 「みんなでやれる」の信念 | 地域完成度メーター（「静岡市の全員で234種」）。個人ランキングではなくチームの成果 | 未実装（地域メーターなし） | ❌ |
| **自己決定理論** (Deci & Ryan) | 自律性・有能感・関係性の3欲求 | 下記で詳述 | 下記参照 | 🟡 |
| **ジョブ・クラフティング** | 自分の役割を自ら意味づけ直す | プロが「ただの査読者」→「ikimonの同定支援者」として新しいアイデンティティを獲得 | 未実装（プロ招待フローなし） | ❌ |

#### 自己決定理論（SDT）の3欲求 × ikimon

| 欲求 | 定義 | ikimonでの設計 | ❌ やってはいけないこと |
|---|---|---|---|
| **自律性** | 自分で選んでいる感覚 | 投稿も同定も任意。確信度も「たぶん」でOK。強制しない | ❌ 「種名を入れないと投稿できません」 |
| **有能感** | できるようになっている感覚 | 同定ガイドで正答に導く。レッドリスト表示で「希少種を見つけた！」 | ❌ 「あなたの同定は間違いです」 |
| **関係性** | つながっている感覚 | プロからの修正が「新しい情報」として届く。地域の仲間と地図を埋める | ❌ プロと初心者を分断する序列 |

#### ジョブ・クラフティングの具体例

| ユーザー | 既存の役割 | ikimonで獲得する新しい意味 |
|---|---|---|
| 退職した生物教師 | 「もう現役じゃない」 | 「地域の同定支援者として頼られている」 |
| 企業のCSR担当 | 「報告書を書くだけ」 | 「地域の生物多様性を可視化するプロジェクトリーダー」 |
| 散歩好きの主婦 | 「趣味で写真を撮ってるだけ」 | 「地域の生態地図を埋める市民科学者」 |
| 環境コンサルタント | 「受注した調査をこなす」 | 「ikimon代理店として地域のデータインフラを構築する」 |

> **設計のリトマス試験**: 新機能を追加するとき「これはユーザーの自律性・有能感・関係性を高めるか？」と問う。3つとも満たさないなら実装しない。

### 📮 投稿のしやすさ

#### ペルソナ別ニーズ
| ペルソナ | 状況 | 求めること |
|---|---|---|
| 親子で散歩 | 子どもが虫を見つけて「撮って！」 | 5秒以内にカメラ起動→撮影→完了 |
| 散歩中の一般人 | 珍しい鳥がいた。逃げる前に撮りたい | ギャラリーからの後出し投稿もOK |
| プロのフィールド調査 | 1トリップで20種記録したい | 一括投稿 or 連続投稿のストレスなし |
| 自然観察会の講師 | 参加者に「ここで投稿してみて」と指示 | URL共有→即投稿。アカウント不要の投稿経路？ |

#### 投稿パターンの全網羅
| パターン | 状況 | 位置情報 | 日時 | 設計上の注意 |
|---|---|---|---|---|
| **A. カメラ直撮影** | ikimon上でカメラ起動→撮影 | ブラウザGPS | 撮影時刻自動 | 最も理想的。PWA連携必須 |
| **B. ギャラリー選択** | 過去に撮った写真をアップ | Exifから抽出 | Exifから抽出 | 「いつどこ？」を聞く必要がある場合あり |
| **C. Share-to-ikimon** | 他アプリの共有メニューから送信 | Exifから抽出 | Exifから抽出 | PWA Share Target API。実装難度高 |
| **D. LINEやSNSで送られた写真** | 友人から「これ何？」と転送された | **なし** | **なし** | Exif消失が多い。手動入力UIが必須 |
| **E. 一括投稿** | プロが1トリップの全記録をまとめて投稿 | Exifから一括抽出 | Exifから一括抽出 | ドラッグ&ドロップ or フォルダ選択 |
| **F. 写真なし観察** | 「鳥の声が聞こえたが撮れなかった」 | ブラウザGPS | 入力時刻 | 音声投稿？テキストのみ？検討必要 |
| **G. スクリーンショット** | SNSで見かけた生物（他人の写真） | **なし** | **なし** | 著作権問題。投稿不可 or 「参考」タグ |

#### Exif欠損時のフォールバック
端末によっては日時・位置情報がExifに記録されない。その場合の段階的フォールバック：
1. **第1優先**: Exifから自動抽出（最も正確）
2. **第2優先**: ブラウザGeolocation API で「今ここ」を取得
3. **第3優先**: 地図タップで位置を指定（「だいたいこの辺」）
4. **第4優先**: 市区町村名からの自動変換
5. **最終手段**: 「不明」で投稿OK（後から追記可能）

> **設計原則**: どの段階で止まっても投稿は完了する。位置不明・日時不明でも投稿は拒否しない。

#### 投稿フローの設計原則
1. **種名は後から**: 投稿時に名前が分からなくて当然。写真＋位置＋日時だけで完結
2. **写真は複数枚**: 同定精度を上げるなら「全体」「拡大」「裏側」が欲しいが、**強制しない**
3. **GPSは自動**: 手入力は最終手段。上記フォールバック順に段階的に対応
4. **ギャラリー投稿対応**: その場で撮れなかった場合、後でカメラロールからアップ
5. **投稿後の即時フィードバック**: 「投稿されました！」+ 地図上にピン表示で「自分の足跡」感
6. **Exifなし端末を排除しない**: どんな端末からでも投稿可能。完璧なデータより「投稿してくれた事実」が価値

| **実機テスト未実施** | 🔴 | カメラ連携・GPS取得の挙動が未検証 |

#### 投稿フォーム三分類（必須 / 任意 / 自動）

> **原則**: 「3タップ感」を実現するには、フォームの**見た目の軽さ**が命。任意フィールドはデフォルト折りたたみ。

| フィールド | 分類 | Phase 0 実装 | 補足 |
|---|:---:|:---:|---|
| 写真（1枚以上） | **必須** | ✅ | 複数枚もOKだが1枚で投稿完了 |
| GPS座標 | **自動** | ✅ | Exif→ブラウザGPS→手動の段階的フォールバック |
| 日時 | **自動** | ✅ | Exif→現在時刻 |
| 種名 | **任意** | ✅ | 後から付与可能。`menot`でデフォルト投稿 |
| メモ | **任意** | ✅ | 折りたたみで隠す |
| 写真追加（複数枚） | **任意** | ✅ | 「+追加」ボタン。初期表示は1枚のみ |
| 環境メタデータ | **任意** | 🟡 | フォームに存在するが未使用。**折りたたみに移動** |
| サイト紐付け | **自動** | ✅ | GPS座標からSiteManagerで自動判定 |

> **アクション**: `post.php` の環境メタ等の非必須フィールドを**デフォルト折りたたみ**にして「3タップ感」を演出。必須は写真だけ、あとは自動。


#### 競合比較
| アプリ | 投稿ステップ数 | 特徴 |
|---|:---:|---|
| iNaturalist | ~5ステップ | 写真→位置→日時→種名候補（AI）→投稿 |
| Biome | ~3ステップ | 写真→AI判定→投稿。ゲーム的 |
| **Geographica** | **投稿機能なし** | **GPSトラッキング特化**。調査員がルート記録に使用 |
| **ikimon (目標)** | **3ステップ** | 写真→GPS自動→送信。**GPSトラッキング内蔵でGeographica不要に** |

> **🎯 Geographica置換戦略**: 調査員は現在「Geographicaでルート記録 + 別アプリで写真記録」の2ツール運用。ikimon がGPSログ + 写真投稿を統合すれば「こっちのほうがいいじゃん！」の勝ち筋。要素10のPWAフィールド体験と直結。

### 🔰 同定しやすさ（初心者）

#### ペルソナ
- 生き物が好きだが知識は図鑑を数冊読んだ程度
- 「チョウ」と「ガ」の違いくらいは知ってる
- 専門用語（「腹板」「前胸背板」等）を言われてもピンとこない

#### 同定の感情ジャーニー
```
発見！ → 撮った！ → これ何だろう？ → 調べてみよう
  😊        📸         🤔              🔍
          ↓
  ガイドが始まる → 質問に答える → 分岐を進む → 候補が出た！
       😮             🤔          😬/😊        🎉
          ↓                         ↓
  用語が分からない…            行き詰まった…
       😰                        😞
          ↓                         ↓
  ポップアップ解説で理解!      「分からなくても投稿OK！
       😊                      誰かが教えてくれるかも」
                                    🤝 ← ここが🎓プロとの接点
```

#### 設計原則
1. **専門用語には必ず視覚的ポップアップ**: テキストだけでなく「ここを見て」写真付き
2. **検索表にない生物**: 「見つかりませんでした」で終わらせない。→「分からないまま投稿 → プロに聞く」への自然な退避路
3. **共通種ショートカット**: スズメ・アゲハチョウなど超メジャー種は検索表を辿らなくてもワンタップで選べる
4. **確信度の選択**: 下記の4段階。初心者が「間違えたらどうしよう」と怯えない設計

##### 確信度（confidence）仕様 — 実装済みコード値との対応
| コード値 | 表示ラベル | 同定ガバナンスLv | 付与タイミング |
|---|---|:---:|---|
| `menot` | （表示なし） | Lv.0 | 投稿時デフォルト（種名未入力） |
| `maybe` | たぶん | Lv.1 | ユーザーが「たぶんこれ」で同定 |
| `sure` | 自信あり | Lv.1-2 | ユーザーまたはプロが確信を持って同定 |
| `literature` | 文献確認済み | Lv.3 | 書籍出典付きで同定 |

> **実装箇所**: `id_form.php`, `observation_detail.php`, `api/post_identification.php`, `api/post_observation.php`
5. **同定の「やり直し」**: 後から修正できることを明示。一発勝負ではない安心感

#### 現在のギャップ
| ギャップ | 重要度 | 説明 |
|---|:---:|---|
| **行き詰まり時のフォールバック** | 🔴 | 検索表で答えに辿り着けなかった時の出口がない |
| **用語の視覚解説** | 🔴 | 専門用語がそのまま表示されると初心者は離脱する |
| **確信度の概念** | 🟡 | 「たぶん」投稿を許容する仕組みがない |
| **共通種ショートカット** | 🟡 | メジャー種にわざわざ検索表を辿らせるのは無駄 |
| **同定の修正フロー** | 🟡 | 一度付けた種名を後から変更する UI が未整理 |

### 🎓 同定しやすさ（プロ）

#### ペルソナ
| 種別 | 具体例 | ikimon への期待 |
|---|---|---|
| 分類学者 | 大学の甲虫研究者 | 自分の同定をデータとして蓄積 + 引用の正確性チェック |
| 環境コンサルタント | 生物多様性調査の実務者 | フィールド記録の効率化 + レポート用データ出力 |
| 生物マニア | 特定分類群の在野専門家 | 他者投稿の同定支援 + 地域データへのアクセス |
| 自然観察指導員 | 観察会を主催する人 | 参加者の投稿をまとめて管理 + 教育素材として利用 |

#### プロが ikimon を選ぶ理由（競合優位）
1. **Citation-First**: 自分の出版物・著述が正しく引用される ← iNaturalist にはない
2. **Anti-AI**: 「AIに同定させて終わり」ではなく、人間の専門性が尊重される

##### Anti-AI Policy — 境界線の定義

| レベル | 許容 | 例 | 根拠 |
|---|:---:|---|---|
| 大分類の自動タグ付け | ✅ | 「これは昆虫です」→ ID Wizard開始点の省略 | 教育的価値なし。効率化OK |
| 検索表ナビゲーション支援 | ✅ | LLMが「翅の模様を見て」とガイド | 人間が判断し、AIは道案内 |
| **種レベルの同定判断** | ❌ | AIが「アゲハチョウ」と断定 | **ikimonの存在意義の否定** |
| AI同定結果の参考表示 | 🟡 | 「Google Lensではこう出ました（参考）」 | 便利だが「正解を先に見せる」弊害 |
| 写真品質の自動判定 | ✅ | 「ピンボケです。撮り直しますか？」 | 同定精度を上げる補助 |

> **判断基準**: 「AIが結論を出す」→ ❌ / 「AIが人間の判断を支援する」→ ✅

3. **書籍のスーパーインデックス**: 自分の専門分野の文献にすぐアクセスできる
4. **地域密着データ**: 日本の地域生物相に特化し、和名が第一言語

#### 現在のギャップ
| ギャップ | 重要度 | 説明 |
|---|:---:|---|
| **同定支援フロー** | 🔴 | 他人の「これ何？」投稿にプロが同定を付ける仕組みがない |
| **専門性の可視化** | 🔴 | 「鳥類に強い人」「甲虫専門」のような分野バッジがない |
| **一括投稿** | 🟡 | 1トリップ20種を1件ずつ投稿するのは非現実的 |
| **同定の検証/修正** | 🟡 | 既存同定に「これは○○では？」とコメントする機能がない |
| **データ利用ポリシー** | 🟡 | 論文にデータを引用する際のライセンス（CC BY 等）が不明確 |
| **分類群ツリーブラウズ** | 🟡 | 「昆虫 > 鞘翅目 > ○○科」で投稿を絞り込めない |
| **書誌データ補正** | 🟢 | Citation の誤りをプロが修正提案できるフロー |

> **重要**: 「同定支援フロー」を要素5（コミュニティ温度）のサブ機能として仕様化する。

#### 同定支援フロー仕様（要素5のサブ機能）

投稿者とプロの接点を設計する。ikimon最大のギャップ。

```
投稿(Lv.0 未同定) → 「🔍 同定募集中」表示
  ↓
プロの通知フィードに表示（分類群フィルタ付き）
  ↓
プロが同定提案(Lv.2/3) → api/post_identification.php
  ↓
投稿者に「新しい情報が見つかりました」通知
  ↓
異論がある場合 → 並列表示（「Aさん: XX / Bさん: YY」）
  ↓
上位プロの裁定 or 出典付き確認で決着
```

|---|---|:---:|
| プロ向け「未同定一覧」フィード | `api/get_pending_identifications.php` [NEW] | 🔴 |
| 同定提案の通知トリガー | `api/post_identification.php` 拡張 | 🟡 API存在、通知なし |
| 分類群フィルタ（「甲虫だけ見たい」） | プロプロフィールに専門分野設定 | 🔴 |

### 🤝 ユーザーとの距離感

#### 距離感の3層モデル
| 層 | 定義 | 現状 |
|---|---|---|
| **第1層: 初接触** | 「ここはどんな場所？安心できる？」 | `onboarding_modal.php` 未活用。第一印象が未設計 |
| **第2層: 日常接点** | 「投稿したら誰かが見てくれてる？」 | 通知・いいねは仕組みはあるがコールドスタート |
| **第3層: 帰属意識** | 「ここは自分の居場所」 | ランキング・バッジはあるが「仲間感」の演出がない |

#### トーン設計（プラットフォームの「声」）
| 場面 | ❌ 冷たい例 | ✅ 温かい例 |
|---|---|---|
| 投稿成功 | 「投稿が完了しました」 | 「新しい足跡が地図に刻まれたよ！🐾」 |
| 空のタイムライン | 「投稿がありません」 | 「最初の発見者になろう！近くにまだ記録のないエリアがあるよ」 |
| 同定失敗 | 「一致する種が見つかりませんでした」 | 「まだ謎の生き物... 誰かが気づいてくれるかも！」 |
| 長期離脱後 | （何もなし） | 「おかえり！前回の○○の投稿エリアで、新しい発見があったよ」 |
| エラー | 「Error 500: Internal Server Error」 | 「ごめん、ちょっと不調。もう一度試してみてね」 |

#### 季節性への対応
生き物は季節で変わる。冬場に「今日の発見0件」が続くと離脱する。
- **冬**: 冬鳥・冬虫夏草・樹皮観察などの「冬ならではのミッション」提案
- **梅雨**: 「雨の日の生き物」特集
- **春/秋**: 自然にアクティブな時期 → 新種発見チャレンジ

#### 地域エンゲージメント設計（単純ランキングの代替）

**コンセプト: 「生態地図の完成度」**

単純な投稿数ランキングは **序列を作る** → 一部のパワーユーザーが独壇場 → 他の人がやる気を失う。
ikimon は代わりに **「みんなで地図を埋める」** モデルを採用。

```
┌─────────────────────────────────────────┐
│  静岡市の生態地図  完成度: 47%          │
│  ████████████░░░░░░░░░░░░  234種 / 推定500種  │
│                                         │
│  🆕 今月 +12種   📈 前月比 +5.2%         │
│                                         │
│  近くのエリアと比べてみよう:             │
│  浜松市  39% ████████░░░░░░░░░░░░░      │
│  藤枝市  31% ██████░░░░░░░░░░░░░░       │
│  焼津市  28% █████░░░░░░░░░░░░░░░       │
└─────────────────────────────────────────┘
```

| 設計要素 | 仕組み | 効果 |
|---|---|---|
| **完成度メーター** | 確認種数 / 地域の推定種数 | 「まだ見つかってないのがある！」の探究心 |
| **近隣エリア比較** | 隣接市区町村 or 気候的に近い地域を並列表示 | 競争ではなく「隣も頑張ってるな」の刺激 |

| **未発見シルエット** | まだ記録のない種を影で表示 | 「これを見つければ地図が埋まる」のゲーム性 |
| **集団達成** | 「静岡市の全員で234種」と表示 | 個人の序列ではなくチームの成果 |
| **スケール切替** | 町内会 / 市区町村 / 都道府県 / 地方 | 自分の活動圏に合わせて拡縮 |

**なぜこれが単純ランキングより良いか:**
- ゼロサムではない（誰かの投稿は全員の完成度を上げる）
- 初心者の1投稿もプロの1投稿も同じ「1種追加」
- 少数ユーザーのエリアでも「未発見が多い＝伸び代」としてポジティブに見える
- B2B&G的にも「この地域は47%把握済み」は説得力のある営業データ

#### 同定ガバナンスモデル（🔰 と 🎓 の両立設計）

**核心の緊張関係**: プロの同定が軽く扱われると無力感で離脱。初心者が怯えると投稿されない。

##### 同定の重み付けシステム
| レベル | 付与条件 | 同定の安定性 | 表示 |
|---|---|---|---|
| **Lv.3: 文献確認済み** | 🎓プロが同定 + 出典付き | 🔒 最も覆りにくい。覆すには同等以上の出典が必要 | ✅ 文献確認バッジ |
| **Lv.2: プロ確認済み** | 🎓プロが同定（出典なし） | 🔐 安定。別のプロが異論を出した場合のみ再検討 | ☑️ プロ確認バッジ |
| **Lv.1: ユーザー同定** | 投稿者が「自信あり」で同定 | 🔓 プロの確認待ち。プロが修正提案可能 | （バッジなし） |
| **Lv.0: 未同定** | 種名なし or 「分からない」 | ⬜ 誰でも同定を提案できる | 🔍「同定募集中」 |

**プロの無力感を防ぐ仕組み:**
- Lv.3 の同定は **同等の文献根拠がない限り覆らない**。「素人の多数決でプロが負ける」は起きない
- プロの同定実績が専門分野バッジとして可視化される（「鞘翅目 120件確認」等）
- プロの修正提案は通知で投稿者に届くが、**Lv.2以上のプロ同定は即反映**（投稿者の承認不要）

**ライト層に優しい仕組み:**
- Lv.0（未同定）で投稿しても **一切ペナルティなし**。むしろ「同定募集中」として目立つ
- 同定を間違えても **履歴として残るだけで恥ずかしくない**。「これは学びの記録」
- プロからの修正は「あなたが間違い」ではなく「新しい情報が見つかりました」のトーン
- 確信度を「たぶん / 自信あり / 文献で確認済み」の3段階で選べる → 「たぶん」が正当な選択肢

**過去の同定がひっくり返りにくい設計:**
- **上位レベルの同定は下位レベルでは覆らない**（Lv.3 は Lv.1 の多数決では変わらない）
- 同レベル間の異論は「並列表示」（「Aさん(Lv.2): XX / Bさん(Lv.2): YY」）→ 上位の確認待ち
- 分類学的改訂（学名変更等）はシステム側で一括反映。個別修正ではない

#### データ品質ガバナンス
| グレード | 定義 | 仕組み | 統計利用 |
|---|---|---|:---:|
| **Grade A: 文献確認** | Lv.3 同定 | 出典＋プロ確認 | ✅ 完全利用 |
| **Grade B: プロ確認** | Lv.2 同定 | プロの目視確認 | ✅ 利用可 |
| **Grade C: ユーザー同定** | Lv.1 同定 | 自己申告+自信あり | ⚠️ 参考値として利用 |
| **Grade D: 未確認** | Lv.0 | 種名なし | ❌ 統計から除外、地図には表示 |

#### 誤同定の扱い
- **トーン原則**: 「別の可能性もあるかも」であり「間違ってる」は禁止
- プロ(Lv.2+)の修正 → 即反映 + 投稿者に「新しい同定情報」として通知
- ユーザー間の異論 → 並列表示 + プロの判定待ちキューに入れる
- 修正履歴を完全保持。学名変更の経緯が追えるようにする
- **「修正された」のではなく「知識がアップデートされた」** と表現する

- 営業・派遣・サポートを代理店に委託 → ikimonはコード書くだけ

- 環境コンサルにとっては「既存クライアントに追加提案できるツール」→ Win-Win

##### 市場独占（一人ユニコーン）の4つの堀

1. **データの堀**: ikimonに蓄積されたモニタリングデータは他ツールに持ち出せない。長く使うほどロックイン
2. **ネットワークの堀**: 市民観察者が増えるほどデータの密度が上がる → 他サービスでは再現不可能
3. **制度接続の堀**: 「自然共生サイトの○%がikimonを使用」→ 環境省の参考ツールに → 事実上の独占
4. **レッドリスト統一DBの堀**: 47都道府県のバラバラなレッドリストを統一UI+APIで提供。**これだけで来る価値がある**

##### レッドリスト統一DB（🛡️🏛️の両方に効く武器）

**現状の問題:**
- 環境省版レッドリスト + 47都道府県版 = 合計48種類のリストが存在
- カテゴリ表記が微妙に異なる（独自カテゴリを設ける県もある）
- `jpnrdb.com`（全国レッドデータ検索システム）があるがUXが壊滅的
- 種名の表記揺れ（和名・学名の不一致）が大量にある
- 「この種はこの県でレッドリスト入りしてるの？」が即座に分からない

**ikimonの解:**
- 47都道府県のレッドリストを統一スキーマでDB化
- 種名はGBIF Backbone Taxonomyで正規化（分類名一致問題の解決）
- 投稿時に「⚠️ この種は静岡県で準絶滅危惧です」が自動表示
- レポート生成で「敷地内にレッドリスト掲載種が○種確認」が自動算出

**スコープ制限（重要）:**
- ❌ **世界規模は現時点では対応しない**。IUCNとの分類名マッチングは地獄
- ⭕ **日本国内47都道府県に集中**。これだけで十分な競合優位
- 月次で更新したい → 都道府県の更新頻度を追跡し、差分をインジェストするパイプライン

| 対象 | 効果 |
|---|---|
| 一般ユーザー | 自分の投稿が「実はレアな種だった！」の発見 |
| 代理店（環境コンサル） | クライアント報告書に「レッドリスト種○種確認」が自動で入る |
| B2B&Gクライアント | TNFD開示・自然共生サイト認定に必要な保全対象種リスト |
| ikimon自身 | **他のどのプラットフォームにもない独自データ** |

##### Geographica置換が独占を加速する理由
- 現在、調査員はルート記録にGeographicaを使っている
- ikimonがGPSトラッキングを内蔵 → 「調査用GPS + 記録アプリ + レポート生成」が1つに
- **調査員（代理店側の人材）がikimonを使い始める** → データの質が上がる → 営業力が上がる
- 調査員向けには無料（Tier 0相当）でOK。彼らが生む高品質データこそが商品

#### 営業ストーリー（3段論法）
1. **課題の提示**: 「御社はTNFD開示で生物多様性データの開示を求められていませんか？」
2. **ikimon の解**: 「市民科学 × 書籍引用の二重検証データを、御社敷地に特化して提供できます」
3. **証拠**: 「741冊の引用元 + GBIF準拠 + Darwin Core エクスポート。学術的に筋の通ったデータです」

#### 観察会 × B2B&G の棲み分けモデル

**原則**: 観察会 = ユーザー獲得チャネル（無料）。B2B&G = 蓄積データを収益化（有償）。競合ではなく漏斗型。

```
観察会（無料）          B2B&G（有償）
│                        │
│  一般参加者が           │  企業・自治体が
│  楽しんで投稿           │  データに予算を出す
│         │              │          │
│         ▼              │          ▼
│  投稿データが蓄積  ───▶  蓄積データでレポート生成
│         │              │          │
│         ▼              │          ▼
│  地域の完成度UP      │  TNFD/30by30準拠のPDF
│   （全員の成果）       │   （クライアント納品）
└────────────────────────┘
```

**競合しない設計のポイント:**
- 観察会データ = **公開データ**（地域完成度に貢献）。誰でもアクセス可能
- B2B&Gデータ = **敷地特化のプライベートデータ** + 自動レポート → 付加価値が違う
- 観察会の蓄積データはB2B&G提案時の「この地域は○%把握済み」の営業素材
- 企業が観察会をスポンサー → CSR + データ収集の一石二鳥
- 観察者派遣（Tier 3）は観察会ノウハウの延長 → 新たな人材採用不要

## スケーラビリティ設計原則（10M UUへの道）

> **目標**: 将来的に**1,000万UU + 高同時接続**に耐えるアーキテクチャ。
> **現実**: 今すぐ作る必要はない。**でも今の設計判断で移行を不可能にしてはいけない**。

### アーキテクチャ進化パス

| Phase | UU | インフラ | データ | 移行トリガー |
|---|---:|---|---|---|
| **0（現行）** | ~1,000 | 共有ホスティング (PHP) | JSON files | — |
| **I** | ~10万 | VPS or Cloud Run | **PostgreSQL** + JSON → DB移行 | 書き込みが秒間10+超えたら |
| **II** | ~100万 | Kubernetes / Cloud Run auto-scale | PostgreSQL + **Redis** + **CDN** | レスポンス1秒超えたら |
| **III** | **1,000万** | **エッジコンピューティング** + マイクロサービス | 分散DB + Object Storage + Search | MAU 50万超えたら |

### Phase 0 → I の移行を楽にする「今守るべき原則」

| 原則 | 具体的なルール | 理由 |
|---|---|---|
| **ステートレスAPI** | サーバー側にセッション状態を持たない | 水平スケーリングの前提 |
| **標準フォーマット** | GeoJSON / Darwin Core / JSON Schema | どのDBにも移行できる |
| **クライアントヘビー** | Alpine.js + MapLibreで描画処理をクライアントに寄せる | サーバー負荷を最小化 |
| **静的ファースト** | レポートPDF・図鑑ページは事前生成。リクエスト時に動的生成しない | CDNキャッシュ可能 |
| **データとロジックの分離** | `DataStore.php` のCRUD抽象化を徹底 | DB移行時にストレージ層だけ差し替え |
| **写真は外部ストレージ** | 写真ファイルはサーバーローカルに置かない設計に早めに移行 | Object Storage (S3/GCS) への移行が楽 |

### ❌ 今やってはいけないこと（移行地獄の原因になる）

| アンチパターン | なぜダメか |
|---|---|
| PHPのセッション変数にユーザー状態を保存 | 複数サーバーで共有できない |
| ファイルパスをハードコード | インフラ変更で全部壊れる |
| SQLなしで複雑なクエリをJSON走査で実装 | DB移行後に二重実装になる |
| ファイルロックで排他制御 | 分散環境で動かない |
| 画面HTMLにビジネスロジックを混ぜる | API化できなくなる |

### ⚠️ Phase 0 の自覚的違反一覧

> **現実**: Phase 0 のプロトタイプとして許容範囲だが、「知っててやっている」を明記する。

| 原則 | 違反内容 | 実例 | 許容理由 | 解消トリガー |
|---|---|---|---|---|
| ステートレスAPI | `Auth.php` がPHPセッション依存 | `$_SESSION['user']` | 共有ホスティングで1サーバー | Phase IでJWT/tokenに切替 |
| データとロジックの分離 | `post.php` 47KBにビジネスロジック混在 | Exif処理、バリデーション | MVP最短距離 | Phase 14BでAPI分離 |
| 写真は外部 | `uploads/` がサーバーローカル | 共有ホスティング上 | コスト最小 | Phase IでS3/GCS |
| ファイルロック | `DataStore::save()` が `flock()` 使用 | 書き込み競合対策 | 単一サーバー | Phase IでDB transaction |
| 静的ファースト | レポートがリクエスト時動的生成 | `generate_site_report.php` | データが少ないので単体OK | 投稿100+で事前生成+キャッシュ |

> **教訓**: 違反は許容するが、**その事実を忘れない**。解消トリガーに到達したら即対応。

### ⚔️ 2軸並行開発戦略（Phase 14以降の行動原則）

> [!IMPORTANT]
> Phase 14以降は **「B2B軸」と「UX軸」の2軸で並行開発** する。
> どちらか一方に偏ると、もう一方が停滞してデッドロックする。

```
┌─────────────────────────────────────────────────────────────┐

│   🏛️ B2B軸 (「売れるもの」を作る)                          │

│   └── コミュニティ温度 → 「誰かが見てる」感                │
```

| 軸 | 対応要素 | Phase 14A (完了) | Phase 14B (次) |

- 2軸を交互に進めることで「データ供給 ↔ 収益化」のフライホイールが回る

- ✅ 「もう1サイト追加したい」と言われる
- ✅ 「知り合いの団体にも紹介していい？」と言われる

### 営業フロー（オープンデモ起点）

```
1. 「このURL見てください」（ダッシュボードURL送付）
       ↓
2. 相手が自分で触って「なるほど」と理解
       ↓
3. 「御社の敷地でも同じものが出せます」
       ↓
4. 「エリアを描いて観察データを入れるだけです」
       ↓
5. 契約 → エリア設定 → 即座に稼働
```
