# ikimon サービス設計 v2.0: ゼロベース戦略再設計
## 「持続可能性」「巻き込み力」「科学者の信頼」「マニアの推し」を同時に満たす設計

---

## エグゼクティブサマリー

### 既存サービスの問題点
| サービス | 強み | 致命的な弱点 |
|---------|------|-------------|
| **Biome** | ゲーミフィケーション（クエスト、レア度） | 「大人のポケモン」化 → 科学者から信頼されない |
| **iNaturalist** | Research Grade品質 | 学習曲線が急すぎる → 大衆に広がらない |
| **いきものログ** | 環境省の権威性 | UXが死んでいる → データが死蔵される |

### ikimonの独自ポジション：「第三の道」
**ゲームでも研究ツールでもない。実社会の価値を生む「地域インフラ」。**

---

## 1. 設計の核心原則

### 1.1 哲学：「観察は公共財」
観察を「娯楽」でも「義務」でもなく「公共財」と再定義する。

- **全ての観察には具体的な価値がある。** ポイントでもバッジでもなく、保全活動・地域政策・生物多様性クレジットへの貢献可能性。
- **唯一の正解はない。** 同定は判決ではなく議論。市民の好奇心と科学的厳密さの両立。

### 1.2 ステークホルダーバランス
**重要な洞察**: 「専門家向けに設計し、初心者でもアクセス可能」にする。逆ではない。

---

## 2. 持続可能なビジネスモデル

**目標: 補助金に依存しない自走モデル**

### 2.1 収益レイヤー（MECE）
1.  **データ層（無料 → 有料インサイト）**
    -   生データ（観察、位置情報） = **無料（CC-BY）**
    -   分析レポート（サイト別種リッチネス、外来種アラート等） = **企業・自治体向け有料**
2.  **プラットフォーム層（B2Bサービス料）**
    -   OECM申請サポートツール：エビデンス自動生成
    -   TNFD対応レポート：生物多様性スコアの企業レポート用出力
3.  **コミュニティ層（任意の後援）**
    -   強制サブスクなし
    -   「観察ポイント → 公園にベンチを寄付」：目に見える貢献
    -   Pay-to-Win要素なし。科学的誠実性は譲らない。

### 2.2 コスト抑制（技術アーキテクチャ）
-   **静的ファースト（PHP/JSON）**: DBライセンス不要、サーバー側スケーリングボトルネックなし
-   **クライアントサイド重視**: 画像圧縮、検索フィルタ、地図レンダリングをすべてブラウザで処理
-   **PWAのみ、ネイティブアプリなし**: ストア手数料ゼロ、単一コードベース、QR即時アクセス

## 3. 「簡単すぎない」大衆巻き込み力

**目標: 始めやすく、深掘りしがいがある。**

### 3.1 オンボーディング：摩擦ゼロの入口
-   **ゲスト投稿OK**: 写真を撮って即投稿。最初の3投稿はアカウント不要。
-   **後から紐付け**: 「投稿をライフリストにまとめませんか？」と優しく誘導。
-   **LINEログイン**: 日本人ユーザーに最低摩擦。

### 3.2 エンゲージメントフック（非ゲーム的）
「ポケモン症候群」（意味のないポイント稼ぎ）を避ける。

| やらないこと | やること | 根拠 |
|-------------|---------|------|
| グローバルランキング | **地域達成率**（「浜松市であと3種！」） | パーソナル感。地元への所有意識。 |
| レア度ポイント | **専門家からの一言**（「珍しいですね！」） | 博士の一言は1000いいねに勝る。 |
| デイリークエスト | **季節限定ミッション**（「春の七草チャレンジ」） | 休眠ユーザーを無理なく呼び戻す。 |
| バッジコレクション | **ライフリスト（グリッド表示）** | 埋まっていく快感（ポケモンDNA）。 |

### 3.3 新規ユーザーの「ゴールデンパス」
```
1. 公園入口のQRコード
2. PWAが1秒で起動
3. カメラタップ → 撮影
4. アップロード（Exif自動入力）
5. 「あなたの観察が公開されました！」→ 地図で確認
6. （24時間以内）専門家が「アオスジアゲハですね！」とコメント
7. ユーザーの感想：「誰かが見てくれた。貢献できた。」
```

## 4. 科学的信頼性：専門家に敬意を払われる設計

**目標: 科学者が使いたい・引用したいプラットフォームにする。**

### 4.1 データ品質フレームワーク（iNat成功モデル準拠）

| ステータス | 定義 | 基準 |
|-----------|------|------|
| **Needs ID** | 同定待ち | 写真のみ、種名なし |
| **Casual** | 同定済だが未検証 | 1人の同定、または飼育/栽培個体 |
| **Research Grade** | 科学検証可能 | 2人以上の合意、正確な位置、日付、野生 |

**Biomeとの決定的違い**: 「AIが種を宣言する」なし。人間の合意が王様。

### 4.2 科学者が愛する機能
-   **フルエクスポート（CSV, Darwin Core）**: 実研究に必須
-   **明確なライセンス（CC-BY）**: 論文で引用可能
-   **GBIF連携パス**:（将来）データが世界的な生物多様性DBに流れる

### 4.3 避けるべき機能（科学的信頼を燃やす）
> [!CAUTION]
> -   **種名のAIオートコンプリート**: 誤同定の洪水を生む
> -   **AIベースのレア度スコア**: AIが1回間違えると信頼は永遠に消える
> -   **分類の簡略化**: データを歪める「虫 vs. 昆虫」みたいなショートカットはNG

## 5. マニアの推し：パワーユーザー向け設計

**目標: トップ1%の愛好家が無給の伝道者になる。**

### 5.1 同定センター：専門家の遊び場
-   **「千本ノック」UI**: フロー状態向け設計。1画面1カード。キーボードショートカット（Y/N/Sで同意/異議/スキップ）。ページリロードなし。
-   **分類群フィルタ**: クモだけ同定したい？クモだけ表示。
-   **「博士」バッジシステム**:
    -   分類群ごとの同定精度をシステムが追跡
    -   閾値到達 → 「クモ博士」称号獲得 → プロフィールに表示
    -   「博士」の同定は合意アルゴリズムで重み付け

### 5.2 学習ループ
-   専門家が**「同定メモ」**を追加可能:
    > 「前翅の斑点が5つ、4つではない点に注目。これで類似種と区別できます。」
-   このメモは観察ページに永続し、**生きた教科書**になる。
-   愛好家はこのメモを読んでレベルアップ。

### 5.3 コミュニティダイナミクス
-   **DMなし**: 荒らし・ハラスメントベクトルを削減
-   **「別候補を提案」（「あなたは間違い」ではなく）**: 礼節ある異議をUIに組み込む
-   **観察中心のコメント**: 議論は「データ」について、「人」についてではない

## 7. 差別化サマリー：なぜBiomeのパクリではないか？

| 軸 | Biome | ikimon v2 |
|----|-------|-----------|
| **コアアイデンティティ** | 「自然のソシャゲ」 | 「地域の自然インフラ」 |
| **同定手法** | AIファースト | 人間ファースト（AI=入力補助のみ） |
| **価値単位** | ポイント・レア度 | Research Grade + クレジット可能性 |
| **対象地理** | 全国 | ハイパーローカル（浜松優先）＋深いO2O |
| **持続可能性** | ユーザー課金 + 企業タイアップ | データインサイト販売 + OECM/TNFDツール |
| **専門家との関係** | 疎遠（AIが置き換える） | 中心（彼らがエンジン） |

## 8. ネクストステップ

1.  **専門家と検証**: 2-3人の地元ナチュラリストにこの文書を共有。使うか？推薦するか？
2.  **MVPスコープ定義**: おそらく「写真投稿 → 専門家同定 → ライフリスト」ループのみ。スコアリングもクエストもなし。
3.  **「同定センター」プロトタイプ設計**: 専門家UXが最も難しく重要。まず紙プロトから。
4.  **ローカルパートナーシップ確立**: 公園（例：佐鳴湖）にパイロットQR設置を打診。

## 9. 種データベース戦略（追記）

### 推奨アーキテクチャ
```
ユーザー入力（「アオスジアゲハ」）
      ↓
[1] ローカルキャッシュ（JSON） ← よく使う約10,000種を事前取得
      ↓（ミス時）
[2] GBIF Species Match API ← 主要ルックアップ
      ↓（和名が必要な場合）
[3] iNat Taxa API ← 日本語名補完
```

**ポイント**: GBIFは英語学名に強いが日本語標準和名の網羅性がやや弱い。iNaturalistはコミュニティが和名を追加しているため、**「GBIF + iNat和名補完」のハイブリッド**が現実解。
